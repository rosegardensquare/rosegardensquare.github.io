3：Iterable 接口和 Iterator 方式

这俩都是接口；

Iterator 是 Collection 接口的父接口，代表可以被遍历，提供了 Iterator 接口的遍历方法

这俩不是同时出现的，Iterator 接口出现的早，Iterable 接口后出现；

Iteraable 是代表是否具有迭代功能，Iterator 代表迭代方式，将是否具有迭代功能抽出来，符合单一职责原则。

1：迭代器的原理

迭代器在遍历元素时，内部用指针跟踪元素，首先指针会指向第一个元素的上方；当调用 hashNext 方法时，如果返回 true，

此时调用 next 方法，指针会下移，将当前指针指向谁就返回谁。

6： 快速失败机制

集合中有使用到快速失败机制；

在集合中为了避免并发修改异常，会维护一个 expectedModCount 期望修改次数的字段，表示集合被迭代器预期修改的次数。

还有一个 modeCount 字段，表示实际被修改的次数。在集合被修改的过程中，会去比较这俩值，如果不一致，就会触发快速失败机制。抛出 ConcurrentModifyException，并发修改异常。

当多个线程对结合进行修改时候，会抛出这个异常。

在 增强 for 循环中，对集合中元素增加删除操作，会抛出这个异常；

1：报错原因解析

modeCount 是集合的成员变量，而 expextedModeCount 是Iterator 迭代器内部定义的变量；

在遍历过程中，对结合的增加和删除，只会修改 modeCount 的值，而不会修改 expectedModeCount的值，导致这俩值

不相等。

2：如何避免快速失败异常

  • 因此为了避免快速失败的异常，应该使用迭代器自己的增加和删除方法。

  • 使用 CopyOnWriteArrayList

    它是先把集合拷贝一份，在副本上做修改，add 和 remove 方法都是加锁控制并发。

  • 使用 List集合的 removeIf 方法

3：快速失败引出 CopyOnWrite

1：ConpyOnWrite 的 add/remove 方法都加了锁，为啥还要copy 一份

CopyOnWrite 当只有要 add 或者 remove 的时候，才 Copy 一份去修改，这是一种延时懒惰策略；

它是一种写时复制的容器；

修改完毕，再将原来容器指向 Copy 的容器地址；

它的 add 和 remove 方法加锁，是为了防止在 Copy 的时候，Copy 出来多个副本，导致并发写；

但是它的读操作没有加锁；

这样做的好处是，我们可以并发读，当然这里读取到的数据有可能不是最新的，

因为写时复制的思想是通过延时懒惰更新的策略，来实现数据最终一致性，并非强一致性。

因此，CopyOnWrite 是一种读写分离的思想，读和写不同的容器, 使用于都多写少的场景；

常用于：白名单黑名单，商品类目的访问或者更新场景。

2： 它和线程安全的vector 有啥区别

Vector 始终都是对一个容器操作，读写互斥，同时只能做一件事；
