线程的基础东西

1：并发、并行

并发（Concurrent）:

​ 两个进程，交替使用同一个 CPU。CPU的处理速度很快，用户觉察不出它的停顿，看来其就像同时干很多事情，这其实是cpu时间片技术，并发完成的；

并行：

​ 是对一个操作系统有 1 个以上的 CPU 来说的；当一个cpu 执行一个进程，另一个CPU执行另一个进程，两个进程间互不干扰可以同时进行，成为并行。

2：进程、线程

进程：一个任务就是一个进程，比如：打开一个浏览器就是一个浏览器进程，打开一个记事本，就是一个记事本进程；

​ 进程是分配资源的基本单位；

线程：在一个进程内部，要同时干很多事，就需要同时运行多个子任务，这些子任务叫做线程；

​ 线程是执行任务的基本单位；线程是比进程更轻量级的执行单元；

同一个进程中的多个线程共享资源；

6：线程的状态相关

    新建后还没有调用 start 方法之前

    • 运行

  • blocked 阻塞

3：为啥线程没有 running 状态

cpu 的时间片很短，大概在 10-20 毫秒左右；

也就是说 1s 之内，同一个线程有可能处于就绪状态或者运行状态，如果明确的定义下来，其实状态时不准的；

当我们看到线程时 running 状态，它有可能已经失去cpu时间；

因此，对于线程，我们只需要直到它当前有没有获取时间片就行，这其实就是 runnable 状态；

当一个线程wait 之后，必须等待其他线程调用 notify 或者 notify all 方法，才会从等待队列中移出‘；

被唤醒的这些线程只是进入争夺队列，并不表示立即获得CPU执行权，只是表示它们可以竞争锁，只有当竞争到锁之后才能有机会

ConcurrentHashMap

1：ConcurrentHashMap 的数据结构

1：JDK 1.8 之前

使用分段锁保证线程安全；它将哈希表默认分为 16 个桶，在加锁的时候，针对某一个桶加锁。其他桶不受影响；这样锁的粒度很小；虽然锁力度小，但是高并发下仍然会出现锁竞争；

2：JDK 1.8 开始

采用 cas + synchronized 加锁，也叫锁分离，节点锁的思想；

在添加元素时，该位置上为空，采用 cas 加入元素；

如果该位置上不是空，第一个节点的哈希值是 move，表示当前这个哈希槽正在进行扩容，那么就调用 helpTransFer 方法协助扩容；

​ 如果没有在扩容，那么使用 sync 锁，锁住当前节点后添加元素；

2：为啥 JDK 8 中要采用新的加锁方式

jdk 7 中的分段锁，虽然提高的并发度，但是当段数固定时，并发很高的时候，任然可能导致热点段，出现性能瓶颈；另外，每个段都是独立结构，这可能导致较高的内存占用；

因此，jdk 8 不再使用分段锁，采用的是以桶为维度的节点锁；内部采用大量的 CAS 操作。为的是提高并发减少锁竞争；

3：它为啥使用 Sync 锁而不是 ReentrantLock 锁

我们都知道，大多数情况下，ReentrantLock 锁都比 Sync 锁更好；

在 JDK 1.7 中，ConcurrHas 把哈希表分为多个 Segment， Segment 继承 ReentrantLock，也就是说 Segmrnt 本身就是一个锁。

JDK 8 中是对每个桶加锁，并发冲突小很多，毕竟对同一个桶去同时既要读又要写的概率很低，在并发度不高的场景，Sync 锁不会频繁升级为重量级锁，大多数情况下是在轻量级锁和偏向锁就搞定。

1：put 流程

  • 首先根据 k 计算哈希值，为了使 k 分配的均匀，增加了扰动函数；并经过哈希值与数组长度取模，计算 k 在数组中的位置；

  • 如果该位置上为空，直接就存了；

  • 如果该位置上有元素，那么遍历该位置上的元素，使用 equas 比较 k 是否相等:

    • 如果相等，那么新的 v 覆盖旧的 v;

    • 如果不相等：

      • jdk 8 之前：

        头插法，新元素存数组位置，来的元素挂在它的后面，形成一个链表；

      • jdk 8 开始：

        判断当前位置的数据结构：

        1：如果是链表，那么就尾插法，新元素跟在老元素后面；

        ​ 尾插法之后，要判断是否需要树化：

        ​ 当链表长度 >8 开始树化，如果此时数组长度 < 64 ,不树化，只扩容；

        ​ 否则，树化。

        ​ 当链表长度 = 6 ，把树变成链表。

        2：如果是红黑树结构，封装当前新元素为 TreeNode，加入红黑树中；

  • 一切结束，在判断一次是否需要扩容。

2：get 流程

  • 首先根据 k 计算出所在数组中的位置；

  • 如果该位置上没元素，直接返回 null;

  • 如果有元素：遍历元素：

    • 比较 equals 是否相等，相等的话就直接返回；

    • equals 不相等，判断该结构数据结构：

      • 链表结构

        也是 equals 遍历链表元素

      • 树形结构

        也是

3-3：关于扩容相关的

1：加载因子

加载因子是为扩容服务，决定了每次扩容多少；

2：为啥加载因子是 0.75f

为啥选 0.75，为啥不是1，为啥不是 0.5、0.8 等呢

在 jdk 官方文档中是这样说明的：

0.75 是时间和空间成本的权衡；它是经过大量的数学计算得来的；太小会浪费空间，动不动就要扩容，太大会造成更多的哈希冲突；

3：咋扩容的

首先说为啥需要扩容？

扩容是为了降低哈希冲突的概率；

扩容发生在 put 方法中，当当前元素个数 > 当前容量的 3/4 时，就扩容 2 倍；

扩容会涉及3点：

  • 如果某位置没有形成链表，那么就 rehash ，将元素放到新位置
  • 如果某位置是链表，也将链表重新链接
  • 如果某位置是树，但是链表长度 < 6 ,取消树化，当然 remove 的时候也会判断是否取消树化

3-4：关于初始化相关的

1：初始化容量应该设置多少合适

阿里巴巴开发手册建议初始化的时候设置一个默认容量，

提高性能；

大小 = 默认是 16

或者：大小 = 你需要存几个元素 / 0.75 + 1

4：为啥 jdk 8 开始链表转红黑树

HashMap 解决哈希冲突采用的是链地址法，新元素会增加到链表上，这样就会产生一个问题，当冲突很多，链表有可能会很长，这样，在插入删除和查询的时候，实践复杂度是 On。尽管 JDK 1.8 之后，为了减少哈希冲突， 优化了扰动函数，使不同的 key 尽量分散开，但是效果不是太好；

我们可以想一个问题：当元素过多，应该用那种数据结构来替代链表呢。我们很自然的想到使用二叉树，所谓二叉查找树，一定是 left < root < right，这样我们遍历的时间复杂度从 On 下降到了 O logn。但是极端情况下，二叉查找树会退化成链表。因此我们就需需要二叉平衡树 （AVL树），它在每次插入的时候，会检查左子树和右子树的高度差 <=1，如果 > 1: 就需要左旋或者右旋，使时间复杂度维持在 logn。但是这也不是绝对万无一失的，我们不仅要保证增删的时间复杂度，也要保证查询的时间复杂度。二叉平衡树为了维持高度差 <=1，非常严格，追求绝对的平衡，在每次插入和查找的时候都要左旋或者右旋，消耗时间；因此，对于那些频繁插入和删除的场景，二叉平衡树显然不合适；为了维持插入删除和查询的时间复杂度维持在一个均衡的状态，引入了红黑树。

红黑树的查找、删除、新增的时间复杂度都是 Ologn;

在红黑树中，所有的叶子节点都是黑色的 null, 不存数据。相邻节点不能同时是红色，两个红色之间用黑色间隔。红黑树不会像二叉平衡树那样追求绝对的平衡，它的插入最多旋转 2 次，删除最多旋转 3次；在频繁的插入删除场景中，红黑树效率优于二叉查找树。

5：为啥链表长度= 8 时，才转红黑树？

这个问题引入两个子问题：

1：为啥不在冲突开始的时候就转呢

不立刻转的原因是：从空间维度来说，红黑树的空间占用是链表的2倍，立刻转的话，太浪费空间了，从时间维度上来说，红黑树虽然查找效率高于链表，但是插入没有链表快。每次插入都要旋转和变色，如果 < 8 就立马转，时间和空间没有做到平衡；

2：为啥一定要在 8 的时候才转？

源码中有做解释：

基于泊松分布，官方在做了大量的数学实验后，得出结论，链表长度 = 8 的概率是 0.00000006，这是一个极小的概率；而且红黑树的 TreeNode 节点的内存占用是链表 Node 节点的 2 倍；

3：为啥又在 6 的时候转回来？

这个设计的主要原因是：基于性能和空间的考虑；8 的时候转红黑树，如何小于8就立马转回去，那就导致转换频繁，所以要选择一个 <8 的值，根据官方做的实验，在 6 的时候，红黑树的优势已经不那么明显了。

静态代码块中加载所有属性；

既然是 static 修饰的属性，在类加载过程中就会被初始化，然而 Java 类的加载和初始化过程都是线程安全的；

因此，枚举类天生是线程安全的；

4： 枚举单例如何解决反射破坏

即使单例类的构造方法是私有的，但是还是可以通过反射来破坏它；

枚举的构造方法虽然也是私有的，但是枚举是在类加载的时候加载，而普通方式实现的单例是在运行期调用的时候加载，这是两者的关键区别；因此，枚举不会被反射破坏，而普通方式ide单例会被反射破坏。
