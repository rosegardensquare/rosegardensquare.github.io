哈希冲突如何解决

1：哈希冲突如何解决

有五种方式解决：

  • 开放地址法

    就是一旦发生冲突，就去寻找下一个空地址，这种方式的缺点是导致元素聚集

  • 链地址法

    这也是 HashMap 在处理哈希冲突时的解决办法；

    其实就是将数组和链表结合起来，发挥两者优势；

  • 再哈希法

    当发生冲突后，使用另外一个哈希函数获取地址，直到找到合适的地址

  • 建立公共溢出区

    将哈希表分为基本表和溢出区，冲突的元素放溢出区

  • 一致性哈希

    主要用于分布式系统中，如：分布式缓存，是将数据均分到不同的节点上以减少冲突

6： 快速失败机制

集合中有使用到快速失败机制；

在集合中为了避免并发修改异常，会维护一个 expectedModCount 期望修改次数的字段，表示集合被迭代器预期修改的次数。

还有一个 modeCount 字段，表示实际被修改的次数。在集合被修改的过程中，会去比较这俩值，如果不一致，就会触发快速失败机制。抛出 ConcurrentModifyException，并发修改异常。

当多个线程对结合进行修改时候，会抛出这个异常。

在 增强 for 循环中，对集合中元素增加删除操作，会抛出这个异常；

1：报错原因解析

modeCount 是集合的成员变量，而 expextedModeCount 是Iterator 迭代器内部定义的变量；

在遍历过程中，对结合的增加和删除，只会修改 modeCount 的值，而不会修改 expectedModeCount的值，导致这俩值

不相等。

1：ConpyOnWrite 的 add/remove 方法都加了锁，为啥还要copy 一份

CopyOnWrite 当只有要 add 或者 remove 的时候，才 Copy 一份去修改，这是一种延时懒惰策略；

它是一种写时复制的容器；

修改完毕，再将原来容器指向 Copy 的容器地址；

它的 add 和 remove 方法加锁，是为了防止在 Copy 的时候，Copy 出来多个副本，导致并发写；

但是它的读操作没有加锁；

这样做的好处是，我们可以并发读，当然这里读取到的数据有可能不是最新的，

因为写时复制的思想是通过延时懒惰更新的策略，来实现数据最终一致性，并非强一致性。

因此，CopyOnWrite 是一种读写分离的思想，读和写不同的容器, 使用于都多写少的场景；

常用于：白名单黑名单，商品类目的访问或者更新场景。

零碎的点

1：值传递和引用传递

除了集合、数组、对象会改变最终的值，因为三者是引用传递，其余都不会改变，因为其余的都是值传递，你传的是值过去，里面

2：& 和 &&

&& 有短路功能；

3：抽象类和接口的区别

1：接口

  • 接口的方法默认是 public 修饰；
  • 接口不能有构造方法；
  • 从设计层面来说：接口相当于一种规范；

2：抽象类

  • 一个类只能有一个父类；

  • 从设计层面来说：抽象类相当于一种模板设计；

  • 抽象类可以有构造方法；

  • 抽象类主要是为了复用；我们经常说的模板方法模式；

4 ：成员变量和局部变量的区别

成员变量

​ 属于类变量或者实例；

​ 如果是 sttaic 修饰就是类变量，如果没有static 修饰就是实例变量；

​ 成员变量随着类的创建而存在；

​ 成员变量会自动赋值默认值；

局部变量

​ 属于方法中定义的变量或者方法参数；对象存在堆内存，局部变量也存在栈内存

局部变量随着方法的调用结束而结束；

​ 局部变量不会自动赋值；

5 ： 静态方法和实例方法

静态只能访问静态的，不能访问实例方法；

实例方法没有这方面的限制；

6：final 关键字

能作用的地方

  • 类
  • 方法
  • 变量

7 ：深拷贝和浅拷贝

深拷贝是创建一个一模一样的新对象，新旧对象之间不共享任何内存，新对象的修改不会影响老对象；

浅拷贝，拷贝的是地址；

无论是深拷贝还是浅拷贝，都需要实现 cloneable 接口；并在 clone 方法中重写拷贝的逻辑

8：如何理解多态

多态的概念比较简单：简单来说就是同一个操作作用于不同的对象，可以有不同的解释，产生不同的结果；

如果按照这个理解，多态是运行期的一种状态， Java 的重写是多态的体现，不过也有人提出重写是静态多态的提现，这个问题在 StackOverflow 上有很多讨论，但我更倾向于动态多态；

要实现这个运行期的动态绑定，就要满足 3 个条件：

​ 1：有类的继承或者实现

​ 2：要有重写

​ 3：父类引用指向子类对象

9：重载和重写的区别

1：重写是运行期的概念，只是参数列表不同；重载是编译期的概念，发生在父子类之间

2：运行期绑定，就是说在运行的时候，根据变量的引用指向子类对象，来调用具体的方法；

​ 编译期绑定，就是说在编译期根据参数的不同来决定调用哪个方法；

序列化

1 ：啥是序列化和反序列化

序列化和反序列化是为了解决网络通信中对象的传输问题；

也就是说怎么把当前 jvm 中进程中的一个对象，传输到另一个 jvm 进程中；

序列化的最终目的是：跨平台存储和网络传输，跨平台存储和网络传输的方式就是 IO ;

而 IO 支持的数据格式就是字节数组；

序列化就是将对象转化为二进制字节流；方便存储和传输；

反序列化就是将字节流恢复成对象；

2 ：序列化反序列化原理

序列化的时候 JVM 会调用 ObjectOutPutStream 的 defaultWriteObject 方法，

反序列化的时候，JVM 会调用 InputStream 的 defaultReadObject 方法；

如果类中自己定义了 writeObject 和 readObject 方法，那么 jvm 就不使用默认的方法；

3: 多种序列化的方式

序列化只是定义了拆解对象的方式，那这种方式可以是多种多样的；

常见的序列化方式有：

  • JDK 原生

    只需要实现 Seriable 接口，就可以通过 OutPutStream 将对象转化为二进制的 byte 字节数组

  • ProtBuf

    谷歌推出

  • Hession

    它是一个轻量级的序列化工具，体积更小

  • Kyro

    号称Java对快的

  • JSON

    JSON 序列化工具会生成一串有规则的字符串；

    常见的 JSON 序列化工具有：

    • fastjson
    • jaskon
    • Gson

4： 序列化的技术选型

  • 协议是否支持跨平台
  • 序列化生成的体积
  • 速度

5： 实现序列化和反序列化为啥要实现 SeriableId 接口？

实现那个接口是为了支持序列化和反序列化操作；

ser 接口并没有任何实现方法，只是一个标记接口，这其实是一个规范；

序列化id 的作用：

​ 是为了表示该类可以被序列化，也是为了反序列化的时候做校验

​ JVM 会对比传入的序列化id 是否与本地文件中的id 一致，如果一致才能反序列化；

如果某个字段不想被序列化，使用 transent 修饰

1: 聊一下Java反射机制

java 的反射机制是指：

​ 程序在运行过程当中，可以创建任意一个类的实例、获取类的内部信息；包括：字段、方法、构造器、注解等；

​ 这种动态的获取类的信息的机制叫做反射机制；

​ 反射被视为Java能够实现动态语言的关键，他让Java有了动态性；

2: 反射机制的原理

反射的原理：

​ Java 的程序是这样运行的：java 文件通过编译期编译成 class文件，之后加载到内存，jvm 生成每个类的 java.lang.class 对象；

​ 通过 class 对象，可以获取类的内部信息，并做后续操作；因此，Java 的反射关键是这个 class 对象。

​ 反射是在运行的时候才知道类的信息；

​ 反射是约定大于配置，配置大于硬编码；

3： 反射机制的优缺点

优点：

  • 增加灵活性

    例如：spring 中利用配置解析 bean 的过程，就是利用反射的典型应用场景；

    Sping MVC 的请求调用对应的方法；

    JDBC 的 class.forName() ；

  • 增加复用及拓展性

    动态代理中，使用动态生成的代理类来提升代码的复用性

缺点：

  • 容易出现运行时异常

    相比于显式的接口和方法，编译器能帮助我们进行类型检查；而反射是在运行时才知道类的信息

  • 性能不高

    在运行方法或者获取字段前，需要遍历才能找到

  • 安全问题

    反射可以随意更改字段及方法的访问权限

字符串

1： 字符串概述

1： 字符串的特点

  • 不可变

    字符串其实是一个只读的，一旦初始化后，再去更新时，就是创建一个新的字符串了，然后把新字符串在堆中的地址指向字符串变量；它的 subString 方法、concat 方法、+= 、replace方法都是 new 一个新的字符串；

    它没有提供追加、删除等修改字符的方法；

    intern方法返回常量池中的地址，如果常量池中没有，就在那里创建一个；

    不可变的好处是：多线程访问时候，保证数据的唯一性。

  • final 修饰

    意味着它的方法不能被覆盖

  • 常量池优化

    字符串创建后，会在常量池中分配一个内存空间缓存起来，下次再创建同一个字符串时，直接返回这个缓存的对象。

  • jdk 8 底层是 char[]，从 jdk 9 开始底层是 byte[]。

    这样改的原因是：大多数情况下，字符串存的都是拉丁字符， 拉丁码也就是 IOS-8859-1， 它一个字符占只需要 1 个字节就够了，但是 1 个 char 占 2 个字节，白白浪费了 1 个字节的空间，因此使用 byte 数组节省。如果要是存汉字还是 2 字节，使用 2 byte

2： 字符串为啥要设计成不可变的

字符串是被广泛使用的一种数据结构，设计成不可变是基于多方考虑：

  • 节省堆空间内存
  • 高效
  • 安全，线程安全

2： 字符串常量池

字符串常量池用于存储字符串；

常量池也就是缓存池；

设计思想是为了提高性能和内存开销

3：字符串的 + 操作

其实是一个语法糖；底层是 StringBuilder 的 append 方法

1： 常量 + 常量 （final 变量 + 常量 ）

​ 返回的还是常量池中的

2： 只要有变量 +、final new + 常量

​ 底层调用的是 StringBuild 的 append 的 toString 方法，会 new 一个新的字符串。返回在堆中的地址。

6：StringBuilder 和 StringBuffer

1： 区别

  • 首先，StingBuffer 和 StringBuilder 都是可变的，没有被 final 修饰；

  • StrtingBuffer 线程安全，要记住，它方法加了锁，效率低

    它是可变的

  • StringBuilder 从 jdk 1.5 才开始有，线程不安全，效率高

    它的 toString 方法，底层是 new 的方式

  • 它们底层数组的初始化长度是 16，不够了就调用 Arrays.copyOf() 扩容为之前的 2 倍 + 2 ；

相同情况下，使用 stringbuilder 只比 stringbuffer 效率提升 10%-15% ，但却要冒多线程不安全的风险；

7：intern方法

运行期，intern 方法可以向运行时常量池中增加字符串常量；

当一个string 调用 intern 方法时，会先在运行时常量池中查找是否存在，如果存在就返回这个地址；

如果不存在，就创建一个字面量加入运行时常量池并返回地址；
