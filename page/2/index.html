lock 锁

lock 接口下还有很多实现类。。。

  • ReadWriteLock 读写锁

  • Condition 接口，实现等待通知模式

  • StampedLock 邮戳锁、票据锁，它是优化读写锁的缺点才引入

1：ReentrantLock 用法

由于 Sync 锁很重，JUC lock 包中，引入 ReentrantLock 锁，替代 Sync 锁；

ReentrantLock 内部基于 AQS 实现；

它和 Sync 锁一样，也是一个可重入锁；

2：ReentrantLock 如何获得可重入

它在加锁的的时候，先看下当前获得到锁的线程是不是自己，如果是的话，就可以再次获取到这个锁，只需要简单的把 state 加就行，增加可重入的次数。

当它要释放锁的时候，那就得确保 state = 0 才能释放，加锁几次，解锁也几次。

3：ReentrantLock 的公平和非公平性

默认是非公平，可以通过它的构造方法设置。

1：非公平锁

多个线程不按照申请锁的顺序去获取锁；而是直接尝试 CAS 获取锁，要是获取不到，再进入等待队列不迟；

获取不到锁，就把当前线封装成一个 node 放到队列尾部。这个队列是一个双向队列。

1：代码层面

非公平锁的核心在内部类 NonFairSync ，它实现了 AQS 中的 Sync 类。

2：公平锁

多个线程按照申请锁的顺序去获取锁， 所有的线程都在队列中排队，这样就保证队列中第一个线程先获取到锁。

它进来之后，先看前面有没有前序节点，没有的话，就获取到锁；有的话，就进队列。

1：代码层面

公平锁的核心在内部类 FairSync 中，它继承 AQS中的 Sync。

它跟非公平锁，只有一点差别，那就是在获取锁之前，先判断下自己前面有没有人排队。

3-1：公平锁和非公平锁对比

两种锁分别适用于不同的场景，存在着各自的优缺点。

对于公平锁来说：它优点是，所有的线程都能获取到资源，不会饿死在队列中。但是它会造成吞吐量下降。队列中，没有获取到锁的，会被阻塞，CPU 唤醒线程的开销是最大缺点。

对于非公平锁来说：它可以减少 CPU 唤醒锁的开销，整体吞吐量会高一点，但是会导致队列中一直获取不到锁的线程长时间等待，造成锁饥饿。

  • cas 在修改 volatile 修饰的 state 的时候，成功就代表获取锁
  • 入队列的操作也用到 CAS 。

Synchronized

1：Syn 锁是怎么实现的

sync 是java 中很重要的关键字，主要用来加锁。

无论是修饰方法还是代码块，它们的实现都是要依赖对象的监视器 monitor

它可以修饰代码块、方法。

  • 同步代码块的锁

    使用 monitorenter 和 monitorexit 两个指令实现。可以把 monitorenter 理解为加锁，monitorexit 理解为释放锁；

    每一个对象都有一个计数器，用来记录被锁了几次，初始化值是0，每加一次锁，这个次数就加1。释放锁的时候，这个次数会减去1，直到0的时候，锁就被释放，其他线程后才能去获取锁。

  • 方法级别的锁

    它是隐式的。同步方法的常量池中会有一个 ACC_SYNC 的标志，当一个线程要访问同步方法时，会先去检查这个标志，如果有设置，需要先去获取监视器锁，方法执行完之后才释放锁。这时候其他线程进来，会因为没办法获取监视器锁而阻塞。如何执行方法的时候出现了异常，代码也没有捕获处理异常，那么异常在被抛出来之前监视器会自动把这个锁给释放了。

2：Syn 锁是可重入的

当一个线程获取到了锁，在没有释放锁之前还可以再次获取锁。

3：锁的对象不一样

  • 修饰方法
    • static 方法，那就是类锁
    • 普通方法，对象锁
  • 修饰代码块，括号里的内容
    • 锁的是 class ，就是类锁
    • 锁的是this, 那就是对象锁

4：对象的监视器 monitor

每个对象都有一个监视器。

当去获取锁的时候，就是对这个对象的监视器操作。

1：monitor 的数据结构

在 HotSpot 虚拟机中，monitor 是 ObjectMonmitor.java 对应 c++ 中的 ObjectMonitor.c++。底层是 ObjectMonitor 对象。

它有几个重要属性：

  • _owner：谁获取了 objectMonitor, 它就指向哪个线程
  • _EntrySet：等待获取锁的线程，这些线程都是阻塞状态。
  • _WaitSet：处理waiting 状态的线程
  • 重入锁的次数
  • 获取锁计数器

2：加锁的流程形象比喻

有一个大楼，有一个特殊房间，每次只能有一个人进入，所有的想进去的人，都得在走廊排队，这个走廊就是 entrySet；

有些特殊的已经获准进入的但是暂时无法脱身的人（比如线程挂起），它们会进入一个叫做 wiatSet 的房间，这些人稍后会被叫到重新进入特殊房间。

3：加锁在代码层面分析

当多个线程访问同步代码块的时候，会先进入 extrySet 队列，此时如果某一个线程活得了 objectMonitor 对象后：

  • 它首先会进入一个 Owner 的区域，把owner 属性设置为自己；
  • 设置计数器 + 1

这些弄完之后，就成功获取了锁。

如果这个线程调用了 wait 方法，它就会进入 waitSet 中，会把 owner 设置为 null，把计数器 -1。等待被唤醒。

4：sync 锁很重

jdk 1.6 之前加锁时， sync 会直接调用 ObjectMonitor 的enter 方法，这个操作是重量级的。为啥这么说呢？

因为 java 的自身不能启动线程，它的线程是基于操作系统深入到内核来完成的，应用程序创建线程后，调用 start 方法，此时操作系统就会从用户态切换到内核态，调用 c++ 源码中的 start0 方法，启动线程，因此这个操作很重。

5：锁优化

jdk 6 开始，sync 就做了优化：

  • 引入偏向锁、轻量级锁

  • 锁消除

    是 JVM 的 JIT 编译器对锁的优化，它是 JIT 编译期经过逃逸分析之后，发现加不加锁都不会发生线程安全。

    就是：经过分析，同步代码加不加锁都只会被同一个线程访问。那这时候就会锁消除。

    就比方说：我们经常定义局部变量 StringBuffer , 这时候 JIT 编译器就会帮锁消除。因为 StringBuffer 的方法加了 Sync 锁。

  • 默认开启自旋锁、自适应自旋

    自旋锁是 CPU 空转，没有放弃cpu 时间

5：Sync 能保证原子性、可见性、有序性

  • 保证原子性

    他能保证同一时间只有一个线程去操作共享变量

  • 可见性

    sync 锁在释放锁的时候，会将工作内存中的变量刷新到主内存中。这样，其他线程就能看到了。

  • 有序性

    共享变量只有在全部 sync 包裹中才能保证有序；如果在 sync 作用范围外使用了这些变量，那么它不会有序。

    如果想要严格的保证有序，得使用 volatile。

6 ：mark word

每一个 java 对象头中都有一个 mark word，它记录的是本对象的运行时数据：

  • 对象的哈希码
  • 分代年龄
  • 锁状态

7：Sync 锁升级

  • 无锁

    当一个线程第一次访问同步代码时，JVM 会先在对象头设置当前访问的线程的id，并将状态设置为偏向锁。

  • 偏向锁

    偏向锁模式下，锁总是会偏向第一个获取锁的线程；

    此时，要是有其他线程过来，Jvm 会先检查对象头中偏向锁的标识，如果是自己的线程id，那就直接获取锁成功；如果

    不是自己的线程id：cas 修改偏向锁id，如果修改成功，当前线程获取了偏向锁；如果修改失败，

    就升级为轻量级锁，偏向锁撤销。偏向锁默认是关闭的。

    jdk 15 中偏向锁已经被废除。之前的版本中，jvm 对偏向锁是 4s 之后才启动。如果关闭偏向锁，锁从无锁直接到轻量级锁。

  • 轻量级锁

    有三个动作：

    • 复制 mark word

      JVM 将对象头中 mark word 复制到线程栈中的锁记录（lock record），这个过程叫做 displaced mark word。

      为啥要复制呢？

      ​ 原因是：为了保留对象的原始信息，为了在释放锁的时候能正确的恢复对象头的原始状态。因为锁的获取和释放成对的。

    • cas 更新 mark word

      尝试使用 cas 更新 mark word 为指向锁记录的指针；

      如果这个动作成功，那么这个线程就成功获取轻量级锁；

      如果这个动作失败，表示有其他线程以及获得到了轻量级锁；

    • cas 失败后自旋等待锁释放

      java 8 中轻量级锁的自旋自动开启，之前就引入了只是默认关闭的；

      此时锁会自旋等待锁释放；自旋 15 次之后（JDK 6 之前默认是10次），仍然失败，会升级为重量级锁。

      jdk 6 之后的版本开始，引入了轻量级锁的自适应自旋；

      ​ 它是根据前一次自旋的时间和成功获取锁的概率进行自适应调整。这些都是 C++ 写的。

  • 重量级锁

    升级为重量级锁之后，jvm 会在对象头中记录指向等待队列的指针。

    这种状态下，一个线程想要获取锁，得下进入等待队列阻塞，

    这里的获取重量级锁也有一个自旋：

    ​ jdk 8 中重量级锁自旋默认不开启。

    当锁释放之后，jvm 会从等待队列中选择一个线程唤醒。

    并将这个被选中的线程的状态设置为就绪状态，等待获取 cpu 时间片。

volatile

1：volatile 能保证原子性吗

不能，volatle 可以理解为轻量级锁sync 锁，但是它跟 sync 锁不同的是：它只能修饰共享变量。它只能保证可见性、有序性。

2：volatle 可见性

使用 volatile 修饰变量，写操作时，jvm 会向 cpu 发出 lock 前缀的指令，意思是将工作内存的数据更新到主内存。这样，其他 cpu 会遵守缓存一致性协议，将主内存的数据加载到自己的工作内存，那么其他线程就读到了最新的变量。

因此，它保证了可见性。

1：解决缓存一致性问题

由于 CPU 和主内存之间存在速度差异，为了弥补这个差异，人们在 CPU 和主内存之间加了几个高速缓存，有L1、L2甚至还有 L3；

但是这么多的告诉缓存，也带来了数据不一致的问题。那么久引入了内存模型 JMM。它是一种规范：规定了内存之间共享变量的读写问题。它只是一个理论，实际上解决这个缓存不一致问题的，有两种方案：

1：总线锁

在早期，人们解决这问题是在总线上加锁，保证同一时间只能由一个cpu 访问总线数据，为啥这样做呢，因为CPU个其他元件交互得通过总线。但是这样效率低啊，于是有了第二种方案。

1：总线锁导致总线风暴

锁总线的方案，是使用总线嗅探机制来检测是否有其他 CPU 修改了变量的值，如果有：就将缓存的值更新成新的值。

但是，多个线程频繁的读写操作，会造成大量的总线通信，引发总线风暴。降低了性能。

总线嗅探就是监视总线上的数据传输，以便于了解它传输的数据是否跟自己有关。

2：缓存一致性协议

最著名的是 Intel 的 MESI 协议。

它的核心思想是：当CPU在操作变量时候，发现它操作的是一个共享的变量，那么它会发出信号：其他CPU的缓存中失效，得从主内存取最新的值。

3：volatile 有序性

volatile 可以进制CPU指令重排序，它是通过内存屏障来实现。这保证了代码会严格按照顺序执行。

如：单例模式中的双重检查锁，就是用 volatile 防止指令重排序。

1：内存屏障

它是一种CPU 指令，是 JVM 在java 的class 文件的某些位置，插入这些指令，防止指令重排序。

具体是：

vloatile 写之前：加上 storestore 指令和 loadstore 指令；

volatile 写之后：加上 storeload 和 storestore

volatile 读之后：加上 loadload 和 loadstaore 。

各个操作系统上实现的方式还不一样。

4：JMM相关

1：有了 MESI 为啥还要 JMM

MESI 和 JMM 都是为了解决缓存一致性问题的，但是它们工作的范围和手段不一样。

MESI 是硬件层面的，JMM 是java 软件层面提出的一套规范，而且JMM规范的内容和解决的问题更广。

它们是不同维度的解决方案。

2：JMM

Java 内存模型。

它屏蔽了各种硬件和操作系统的访问差异，规范了工作内存和主内存交互的细节，也规范了如何解决内存有序性。

它不仅有一套规范，还提出了一些java 处理并发的关键字：sync 锁、volatile、fina、concurrent 包，供程序员使用。

现在说的 JMM, 一般都是 JDK 5 开始的新的 JMM。

他具体规范是：

​ 所有共享变量都存在主内存，各个线程有自己的工作内存，工作内存是主内存数据的一个副本，对共享变量只能在自己的工作内存操作，各个线程之间的工作内存独立且不能相互访问。

5：volatile 不能保证 i++ 的原子性操作

i ++ 操作分为三步骤：

  • CPU 拿到主内存的原始值到工作内存
  • 加操作
  • 把工作内存数据写回主内存

那就需要用到原子工具类、或者 sync 锁

6：为啥单例模式需要 volatile

1：new 一个对象的三步骤

  • 给对象分配内存空间
  • 对变量属性初始化
  • 将变量的引用地址写到内存

这就是为啥单例双重检查锁中，需要volatile 禁止指令重排序。

1：啥是线程池，是咋实现的

线程池是一种池化技术，所谓池化技术，就是提前保存大量的资源，以备不时之需；在机器资源有限的情况下，使用池化技术，有效提高资源利用率，提升性能；

线程池，就是提前创建好一批线程，保存在线程池中，当有任务需要执行时，就从线程池中选一个线程去执行，执行完毕再放回去，供其他任务使用；线程池的好处是它能控制并发，而且复用，减少创建和销毁的时间。

2：工作流程

2.1：也叫提交一个任务的流程

当提交一个任务时，先看看核心线程数有没有达到上限；

核心线程数没达到上限，就创建一个新核心的来执行任务；

如何核心满了，就把任务放到等待队列；

如果队列满了，就创建非核心线程从队列中取任务执行；

如果最大的线程数满了，新任务要触发拒绝策略；

当非核心线程数没有处理任务，超过它的空闲时间，就会被销毁。

注意：核心线程也是可以被销毁的。

2.2：使用具体的参数，说明执行顺序

假如：核心线程数是 5，对列长度是100，最大核心数是50，并发是 200，线程池执行顺序是：

前 5 个使用核心线程执行；

6-105 （100个）放入队列；

106-150（45个）非核心去执行；

首先看影响线程池因素有哪些：

  • CPU核心数

    • 多核处理器
    • 超线程技术（英特尔的HT技术），目前，很多CPU都采用超线程技术，也就是利用特殊的硬件指令。把两个逻辑内核模拟成两个物理芯片，让单个处理器都能使用线程级并行计算。所以我们经常看到 4核8线程CPU，就是物理内核4核，逻辑内核8核。可以为每个核心分配更多的线程，因为超线程可以提高CPU利用率。

  • 应用类型

    • CPU密集型

      主要是更多的计算，线程数的公式是：核心线程数 * 1 或者 1.5 ；因为这些主要是消耗 CPU，线程的上下文切换反而影响性能

    • IO密集型

      IO密集型主要涉及到很多的阻塞、等待（比如：数据库操作、文件操作、网络操作）；这样的看可以配置更多的线程，

      设置为 2 倍核心，因为线程需要更长的时间去处理更多的任务。

  • JVM 和内存资源

    • 内存大小
    • 操作系统的线程大小限制

  • 其他考虑

    • RT 要求（响应时间的要求）

      需要更多的线程

    • 任务类型的不同

      长时间的任务、短时间的任务、同步任务或者异步任务

2：有没有公式

根据前面的公式，主要维度是核心数、应用类型；

公式可以作为参考：

  • CPU 密集型

    线程池大小 = N +1

    为啥要加1，是为了预留出某些线程异常的时候

  • IO密集型

    线程池大小 = 2N + 1

3：咋知道是计算密集还是 IO 密集型

还有一个通用公式：

线程数 = CPU 核心数 * （1 + 线程等待时间 / 线程运行总时间）

  • 线程等待时间：

    线程没有使用 CPU 的时候的时间，比如阻塞在了 IO

  • 线程运行总时间

4：不建议直接套公式

我们有时候看到的 CPU 核心数并不一定是真实的；

很多时候，项目部署在云服务器上，有时候分配的机器显示的是8核，但是实际上使用的是虚拟机，并不是物理机；实际上并不能发挥出8核的作用；

可以在刚上线的时候，现根据公式设置一个值，然后再根据实际业务，以及不断的压测，再不断调整；最终达到一个合理的值。

然后在压测的时候，也要说清楚压测可接受的响应耗时是多少，大于这个阈值是错误的，错误率应该多少能接受。这样给出的线程池数量才是合理的。

1：并发、并行

并发（Concurrent）:

​ 两个进程，交替使用同一个 CPU。CPU的处理速度很快，用户觉察不出它的停顿，看来其就像同时干很多事情，这其实是cpu时间片技术，并发完成的；

并行：

​ 是对一个操作系统有 1 个以上的 CPU 来说的；当一个cpu 执行一个进程，另一个CPU执行另一个进程，两个进程间互不干扰可以同时进行，成为并行。

2：进程、线程

进程：一个任务就是一个进程，比如：打开一个浏览器就是一个浏览器进程，打开一个记事本，就是一个记事本进程；

​ 进程是分配资源的基本单位；

线程：在一个进程内部，要同时干很多事，就需要同时运行多个子任务，这些子任务叫做线程；

​ 线程是执行任务的基本单位；线程是比进程更轻量级的执行单元；

同一个进程中的多个线程共享资源；

ConcurrentHashMap

1：ConcurrentHashMap 的数据结构

1：JDK 1.8 之前

使用分段锁保证线程安全；它将哈希表默认分为 16 个桶，在加锁的时候，针对某一个桶加锁。其他桶不受影响；这样锁的粒度很小；虽然锁力度小，但是高并发下仍然会出现锁竞争；

2：JDK 1.8 开始

采用 cas + synchronized 加锁，也叫锁分离，节点锁的思想；

在添加元素时，该位置上为空，采用 cas 加入元素；

如果该位置上不是空，第一个节点的哈希值是 move，表示当前这个哈希槽正在进行扩容，那么就调用 helpTransFer 方法协助扩容；

​ 如果没有在扩容，那么使用 sync 锁，锁住当前节点后添加元素；

2：为啥 JDK 8 中要采用新的加锁方式

jdk 7 中的分段锁，虽然提高的并发度，但是当段数固定时，并发很高的时候，任然可能导致热点段，出现性能瓶颈；另外，每个段都是独立结构，这可能导致较高的内存占用；

因此，jdk 8 不再使用分段锁，采用的是以桶为维度的节点锁；内部采用大量的 CAS 操作。为的是提高并发减少锁竞争；

3：它为啥使用 Sync 锁而不是 ReentrantLock 锁

我们都知道，大多数情况下，ReentrantLock 锁都比 Sync 锁更好；

在 JDK 1.7 中，ConcurrHas 把哈希表分为多个 Segment， Segment 继承 ReentrantLock，也就是说 Segmrnt 本身就是一个锁。

JDK 8 中是对每个桶加锁，并发冲突小很多，毕竟对同一个桶去同时既要读又要写的概率很低，在并发度不高的场景，Sync 锁不会频繁升级为重量级锁，大多数情况下是在轻量级锁和偏向锁就搞定。

1：put 流程

  • 首先根据 k 计算哈希值，为了使 k 分配的均匀，增加了扰动函数；并经过哈希值与数组长度取模，计算 k 在数组中的位置；

  • 如果该位置上为空，直接就存了；

  • 如果该位置上有元素，那么遍历该位置上的元素，使用 equas 比较 k 是否相等:

    • 如果相等，那么新的 v 覆盖旧的 v;

    • 如果不相等：

      • jdk 8 之前：

        头插法，新元素存数组位置，来的元素挂在它的后面，形成一个链表；

      • jdk 8 开始：

        判断当前位置的数据结构：

        1：如果是链表，那么就尾插法，新元素跟在老元素后面；

        ​ 尾插法之后，要判断是否需要树化：

        ​ 当链表长度 >8 开始树化，如果此时数组长度 < 64 ,不树化，只扩容；

        ​ 否则，树化。

        ​ 当链表长度 = 6 ，把树变成链表。

        2：如果是红黑树结构，封装当前新元素为 TreeNode，加入红黑树中；

  • 一切结束，在判断一次是否需要扩容。

  • 首先根据 k 计算出所在数组中的位置；

  • 如果该位置上没元素，直接返回 null;

  • 如果有元素：遍历元素：

    • 比较 equals 是否相等，相等的话就直接返回；

    • equals 不相等，判断该结构数据结构：

      • 链表结构

        也是 equals 遍历链表元素

      • 树形结构

3-1：HashMap 如何确定 k 在数组位置

通过 k 的哈希值取模数组长度；

通过源码发现, HashMap 并不是这种方式，采用了一些骚操作；

它是通过 k 的哈希值右移 16 位，之后跟数组长度 -1 ，做 & 计算；

1： 为啥要用这种骚操作呢

1：为啥要用&不用%

是基于内存的二进制直接计算；比 % 十进制取模更高效；

这也就是为啥它要每次扩容是 2的N次方的原因，它是想用位运算呀

2：为啥要保持数组容量为 2 的N次方

在高效计算 k 在数组中的位置的时候使用；

1：初始化保证

2：扩容保证

3：为啥哈希值要右移 16 位呢

它是为了增加哈希值的扰动，使得哈希分布的尽可能均匀。说白了，就是将hashCode的高16和低16位，进行异或，使得hashCode的值更加分散一点。

3-2：HashMap 如何找 k

先计算出 k 的哈希值，进一步计算出在数组中的位置，然后使用 equals 比较找到 k；

因此，在使用 HashMap 的时候，k 都要重写 hashcode 和 equasl 方法

3-3：关于扩容相关的

1：加载因子

加载因子是为扩容服务，决定了每次扩容多少；

2：为啥加载因子是 0.75f

为啥选 0.75，为啥不是1，为啥不是 0.5、0.8 等呢

在 jdk 官方文档中是这样说明的：

0.75 是时间和空间成本的权衡；它是经过大量的数学计算得来的；太小会浪费空间，动不动就要扩容，太大会造成更多的哈希冲突；

3：咋扩容的

首先说为啥需要扩容？

扩容是为了降低哈希冲突的概率；

扩容发生在 put 方法中，当当前元素个数 > 当前容量的 3/4 时，就扩容 2 倍；

扩容会涉及3点：

  • 如果某位置没有形成链表，那么就 rehash ，将元素放到新位置
  • 如果某位置是链表，也将链表重新链接
  • 如果某位置是树，但是链表长度 < 6 ,取消树化，当然 remove 的时候也会判断是否取消树化

3-4：关于初始化相关的

1：初始化容量应该设置多少合适

阿里巴巴开发手册建议初始化的时候设置一个默认容量，

提高性能；

大小 = 默认是 16

或者：大小 = 你需要存几个元素 / 0.75 + 1

4：为啥 jdk 8 开始链表转红黑树

HashMap 解决哈希冲突采用的是链地址法，新元素会增加到链表上，这样就会产生一个问题，当冲突很多，链表有可能会很长，这样，在插入删除和查询的时候，实践复杂度是 On。尽管 JDK 1.8 之后，为了减少哈希冲突， 优化了扰动函数，使不同的 key 尽量分散开，但是效果不是太好；

我们可以想一个问题：当元素过多，应该用那种数据结构来替代链表呢。我们很自然的想到使用二叉树，所谓二叉查找树，一定是 left < root < right，这样我们遍历的时间复杂度从 On 下降到了 O logn。但是极端情况下，二叉查找树会退化成链表。因此我们就需需要二叉平衡树 （AVL树），它在每次插入的时候，会检查左子树和右子树的高度差 <=1，如果 > 1: 就需要左旋或者右旋，使时间复杂度维持在 logn。但是这也不是绝对万无一失的，我们不仅要保证增删的时间复杂度，也要保证查询的时间复杂度。二叉平衡树为了维持高度差 <=1，非常严格，追求绝对的平衡，在每次插入和查找的时候都要左旋或者右旋，消耗时间；因此，对于那些频繁插入和删除的场景，二叉平衡树显然不合适；为了维持插入删除和查询的时间复杂度维持在一个均衡的状态，引入了红黑树。

红黑树的查找、删除、新增的时间复杂度都是 Ologn;

在红黑树中，所有的叶子节点都是黑色的 null, 不存数据。相邻节点不能同时是红色，两个红色之间用黑色间隔。红黑树不会像二叉平衡树那样追求绝对的平衡，它的插入最多旋转 2 次，删除最多旋转 3次；在频繁的插入删除场景中，红黑树效率优于二叉查找树。

5：为啥链表长度= 8 时，才转红黑树？

这个问题引入两个子问题：

1：为啥不在冲突开始的时候就转呢

不立刻转的原因是：从空间维度来说，红黑树的空间占用是链表的2倍，立刻转的话，太浪费空间了，从时间维度上来说，红黑树虽然查找效率高于链表，但是插入没有链表快。每次插入都要旋转和变色，如果 < 8 就立马转，时间和空间没有做到平衡；

2：为啥一定要在 8 的时候才转？

源码中有做解释：

基于泊松分布，官方在做了大量的数学实验后，得出结论，链表长度 = 8 的概率是 0.00000006，这是一个极小的概率；而且红黑树的 TreeNode 节点的内存占用是链表 Node 节点的 2 倍；

3：为啥又在 6 的时候转回来？

这个设计的主要原因是：基于性能和空间的考虑；8 的时候转红黑树，如何小于8就立马转回去，那就导致转换频繁，所以要选择一个 <8 的值，根据官方做的实验，在 6 的时候，红黑树的优势已经不那么明显了。

6：HashMap、HashTable 、ConcurrentHashMap 比较

1：线程安全角度

HashMap 不是线程安全的；

HashTable 方法加锁，是线程安全；

ConcurrentHs, 也是线程安全的。

2：继承关系角度

HashMap 继承 AsbstractMap 实现 Map 接口；

HashTable 继承古老的 Dictory 接口；

ConcurrentHs 继承 AbstractMap 继承 ConcurrentMap 接口；

3：是否为 null 角度

HashMap 允许 null;

HashTable 和 ConcurrentHs 不允许有 null；

7 ： HashMap 在并发场景会有啥问题

这个问题只是会出现在 JDK 7 之前，JDK 8 开始久修复了

发生在扩容的时候，jdk 7 中，采用头插法，之所以采用头插法，是因为 JDK 作者认为后插入的元素使用频率较高，更大概率成为热点数据；采用头插法把他们放在头部，就可以提高查询效率。

但是，正是因为把当前元素作为链表得到头节点，在并发扩容时，产生循环引用，造成链表死循环。导致 CPU 飙升。

jdk 8 开始的版本，已经解决了这个问题：

jdk 8 开始采用尾插法。

4：啥是 BitSet

位集合的底层结构是一个 bit 数组，如果第 n 位是1，则表示数组 n 在这个集合中；

它是一个可变长度的数组；也就是动态数组；

因为元素有下标，因此根据下标查找速度很快；

但是增加或者删除元素，需要移动后面的元素，所以效率低；

它扩容是 1.6 倍；

线程不安全；

先申请一个容量为 10 的数组，当存入第 11 个元素时候，就扩容为之前的 1.5 倍；并将老数组复制到新数组中完成扩容；
