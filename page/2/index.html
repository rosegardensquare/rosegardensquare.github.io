ConcurrentHashMap

1：ConcurrentHashMap 的数据结构

1：JDK 1.8 之前

使用分段锁保证线程安全；它将哈希表默认分为 16 个桶，在加锁的时候，针对某一个桶加锁。其他桶不受影响；这样锁的粒度很小；虽然锁力度小，但是高并发下仍然会出现锁竞争；

2：JDK 1.8 开始

采用 cas + synchronized 加锁，也叫锁分离，节点锁的思想；

在添加元素时，该位置上为空，采用 cas 加入元素；

如果该位置上不是空，第一个节点的哈希值是 move，表示当前这个哈希槽正在进行扩容，那么就调用 helpTransFer 方法协助扩容；

​ 如果没有在扩容，那么使用 sync 锁，锁住当前节点后添加元素；

2：为啥 JDK 8 中要采用新的加锁方式

jdk 7 中的分段锁，虽然提高的并发度，但是当段数固定时，并发很高的时候，任然可能导致热点段，出现性能瓶颈；另外，每个段都是独立结构，这可能导致较高的内存占用；

因此，jdk 8 不再使用分段锁，采用的是以桶为维度的节点锁；内部采用大量的 CAS 操作。为的是提高并发减少锁竞争；

3：它为啥使用 Sync 锁而不是 ReentrantLock 锁

我们都知道，大多数情况下，ReentrantLock 锁都比 Sync 锁更好；

在 JDK 1.7 中，ConcurrHas 把哈希表分为多个 Segment， Segment 继承 ReentrantLock，也就是说 Segmrnt 本身就是一个锁。

JDK 8 中是对每个桶加锁，并发冲突小很多，毕竟对同一个桶去同时既要读又要写的概率很低，在并发度不高的场景，Sync 锁不会频繁升级为重量级锁，大多数情况下是在轻量级锁和偏向锁就搞定。

​ Entry 数组是内部类，每一个 Entry 包含一个 K、V；数组是主体；

2：遍历 HashMap

3：HashMap 的 put方法、get 方法分析

1：put 流程

  • 首先根据 k 计算哈希值，为了使 k 分配的均匀，增加了扰动函数；并经过哈希值与数组长度取模，计算 k 在数组中的位置；

  • 如果该位置上为空，直接就存了；

  • 如果该位置上有元素，那么遍历该位置上的元素，使用 equas 比较 k 是否相等:

    • 如果相等，那么新的 v 覆盖旧的 v;

    • 如果不相等：

      • jdk 8 之前：

        头插法，新元素存数组位置，来的元素挂在它的后面，形成一个链表；

      • jdk 8 开始：

        判断当前位置的数据结构：

        1：如果是链表，那么就尾插法，新元素跟在老元素后面；

        ​ 尾插法之后，要判断是否需要树化：

        ​ 当链表长度 >8 开始树化，如果此时数组长度 < 64 ,不树化，只扩容；

        ​ 否则，树化。

        ​ 当链表长度 = 6 ，把树变成链表。

        2：如果是红黑树结构，封装当前新元素为 TreeNode，加入红黑树中；

  • 一切结束，在判断一次是否需要扩容。

2：get 流程

  • 首先根据 k 计算出所在数组中的位置；

  • 如果该位置上没元素，直接返回 null;

  • 如果有元素：遍历元素：

    • 比较 equals 是否相等，相等的话就直接返回；

    • equals 不相等，判断该结构数据结构：

      • 链表结构

        也是 equals 遍历链表元素

      • 树形结构

        也是

3-1：HashMap 如何确定 k 在数组位置

通过 k 的哈希值取模数组长度；

通过源码发现, HashMap 并不是这种方式，采用了一些骚操作；

它是通过 k 的哈希值右移 16 位，之后跟数组长度 -1 ，做 & 计算；

1： 为啥要用这种骚操作呢

1：为啥要用&不用%

是基于内存的二进制直接计算；比 % 十进制取模更高效；

这也就是为啥它要每次扩容是 2的N次方的原因，它是想用位运算呀

2：为啥要保持数组容量为 2 的N次方

在高效计算 k 在数组中的位置的时候使用；

1：初始化保证

2：扩容保证

3：为啥哈希值要右移 16 位呢

它是为了增加哈希值的扰动，使得哈希分布的尽可能均匀。说白了，就是将hashCode的高16和低16位，进行异或，使得hashCode的值更加分散一点。

3-2：HashMap 如何找 k

先计算出 k 的哈希值，进一步计算出在数组中的位置，然后使用 equals 比较找到 k；

因此，在使用 HashMap 的时候，k 都要重写 hashcode 和 equasl 方法

3-3：关于扩容相关的

1：加载因子

加载因子是为扩容服务，决定了每次扩容多少；

2：为啥加载因子是 0.75f

为啥选 0.75，为啥不是1，为啥不是 0.5、0.8 等呢

在 jdk 官方文档中是这样说明的：

0.75 是时间和空间成本的权衡；它是经过大量的数学计算得来的；太小会浪费空间，动不动就要扩容，太大会造成更多的哈希冲突；

3：咋扩容的

首先说为啥需要扩容？

扩容是为了降低哈希冲突的概率；

扩容发生在 put 方法中，当当前元素个数 > 当前容量的 3/4 时，就扩容 2 倍；

扩容会涉及3点：

  • 如果某位置没有形成链表，那么就 rehash ，将元素放到新位置
  • 如果某位置是链表，也将链表重新链接
  • 如果某位置是树，但是链表长度 < 6 ,取消树化，当然 remove 的时候也会判断是否取消树化

3-4：关于初始化相关的

1：初始化容量应该设置多少合适

阿里巴巴开发手册建议初始化的时候设置一个默认容量，

提高性能；

大小 = 默认是 16

或者：大小 = 你需要存几个元素 / 0.75 + 1

4：为啥 jdk 8 开始链表转红黑树

HashMap 解决哈希冲突采用的是链地址法，新元素会增加到链表上，这样就会产生一个问题，当冲突很多，链表有可能会很长，这样，在插入删除和查询的时候，实践复杂度是 On。尽管 JDK 1.8 之后，为了减少哈希冲突， 优化了扰动函数，使不同的 key 尽量分散开，但是效果不是太好；

我们可以想一个问题：当元素过多，应该用那种数据结构来替代链表呢。我们很自然的想到使用二叉树，所谓二叉查找树，一定是 left < root < right，这样我们遍历的时间复杂度从 On 下降到了 O logn。但是极端情况下，二叉查找树会退化成链表。因此我们就需需要二叉平衡树 （AVL树），它在每次插入的时候，会检查左子树和右子树的高度差 <=1，如果 > 1: 就需要左旋或者右旋，使时间复杂度维持在 logn。但是这也不是绝对万无一失的，我们不仅要保证增删的时间复杂度，也要保证查询的时间复杂度。二叉平衡树为了维持高度差 <=1，非常严格，追求绝对的平衡，在每次插入和查找的时候都要左旋或者右旋，消耗时间；因此，对于那些频繁插入和删除的场景，二叉平衡树显然不合适；为了维持插入删除和查询的时间复杂度维持在一个均衡的状态，引入了红黑树。

红黑树的查找、删除、新增的时间复杂度都是 Ologn;

在红黑树中，所有的叶子节点都是黑色的 null, 不存数据。相邻节点不能同时是红色，两个红色之间用黑色间隔。红黑树不会像二叉平衡树那样追求绝对的平衡，它的插入最多旋转 2 次，删除最多旋转 3次；在频繁的插入删除场景中，红黑树效率优于二叉查找树。

5：为啥链表长度= 8 时，才转红黑树？

这个问题引入两个子问题：

1：为啥不在冲突开始的时候就转呢

不立刻转的原因是：从空间维度来说，红黑树的空间占用是链表的2倍，立刻转的话，太浪费空间了，从时间维度上来说，红黑树虽然查找效率高于链表，但是插入没有链表快。每次插入都要旋转和变色，如果 < 8 就立马转，时间和空间没有做到平衡；

2：为啥一定要在 8 的时候才转？

源码中有做解释：

基于泊松分布，官方在做了大量的数学实验后，得出结论，链表长度 = 8 的概率是 0.00000006，这是一个极小的概率；而且红黑树的 TreeNode 节点的内存占用是链表 Node 节点的 2 倍；

3：为啥又在 6 的时候转回来？

这个设计的主要原因是：基于性能和空间的考虑；8 的时候转红黑树，如何小于8就立马转回去，那就导致转换频繁，所以要选择一个 <8 的值，根据官方做的实验，在 6 的时候，红黑树的优势已经不那么明显了。

6：HashMap、HashTable 、ConcurrentHashMap 比较

1：线程安全角度

HashMap 不是线程安全的；

HashTable 方法加锁，是线程安全；

ConcurrentHs, 也是线程安全的。

2：继承关系角度

HashMap 继承 AsbstractMap 实现 Map 接口；

HashTable 继承古老的 Dictory 接口；

ConcurrentHs 继承 AbstractMap 继承 ConcurrentMap 接口；

3：是否为 null 角度

HashMap 允许 null;

HashTable 和 ConcurrentHs 不允许有 null；

7 ： HashMap 在并发场景会有啥问题

这个问题只是会出现在 JDK 7 之前，JDK 8 开始久修复了

发生在扩容的时候，jdk 7 中，采用头插法，之所以采用头插法，是因为 JDK 作者认为后插入的元素使用频率较高，更大概率成为热点数据；采用头插法把他们放在头部，就可以提高查询效率。

但是，正是因为把当前元素作为链表得到头节点，在并发扩容时，产生循环引用，造成链表死循环。导致 CPU 飙升。

jdk 8 开始的版本，已经解决了这个问题：

jdk 8 开始采用尾插法。

2：TreeSet 如何保证元素不重复

TreeSet 底层是 TreeMap 的 keySet方法，TreeMap 底层是红黑树，红黑树是一种平衡二叉树；

它能保证任何一个左右子树的高度差不会超过较矮的那颗的一倍。

4：啥是 BitSet

BitSet 是位集合；

位集合的底层结构是一个 bit 数组，如果第 n 位是1，则表示数组 n 在这个集合中；

比如：当调用 BitSet 的 set(10) ，业务的意思是把 10 放到这个集合中，内部的操作是把二进制的第 10 位置为 1，

表示 10 在这个集合中；

在Java 中，它的底层是一个 long 数组，它最大的好处是对于多个数字来说，降低了内存空间，正常情况下，要把所有的 int

类型的正整数，存起来，需要 8G内存，但是使用 bitSet 的话，只需要原来内存的 1/32;

BitSet 常用在哪

  • 现在有一千万个随机数，随机数的范围在 1 - 1亿之间，求出 1-1亿之间的数字，哪些没有在这批随机数中？

  • 将 N 个数字排序

List 接口

1：List 接口总述

有序

可重复

它有索引

这里的有序是指: 存入的元素的顺序和取出时的顺序是否一致。

2：ArrayList

它是一个可变长度的数组；也就是动态数组；

因为元素有下标，因此根据下标查找速度很快；

但是增加或者删除元素，需要移动后面的元素，所以效率低；

它扩容是 1.6 倍；

线程不安全；

1：ArrayList 如何扩容？

先申请一个容量为 10 的数组，当存入第 11 个元素时候，就扩容为之前的 1.5 倍；并将老数组复制到新数组中完成扩容；

2：ArrayList 的序列化时如何实现的？

ArrayList 中重写了序列化和反序列化的两个方法：writeObject 和 readObject

3：LinkedList

它是一个双向的链表；

在 jdk 8 中每个元素是一个 Node;

因为每个元素都记录前后元素的地址，所以它增删元素的时候，只用修改前后的地址就行，增删的效率高；

查找某个元素，得遍历，所以它查找效率有点低；

这些比较都是基于数据量大的时候，数据量小没啥意义。

这家伙还实现了 Queue 和 Dequeue 接口；

4：Vector

初始大小是10，它扩容是 2 倍；

它的所有操作都加了锁，是一个读写互斥的集合

缓存淘汰策略有：

  • LRU

Redis 的 GuavaMap 就是使用这个策略淘汰缓存

  • TTL

淘汰过期的

  • 随机算法

哈希冲突如何解决

1：哈希冲突如何解决

有五种方式解决：

  • 开放地址法

    就是一旦发生冲突，就去寻找下一个空地址，这种方式的缺点是导致元素聚集

  • 链地址法

    这也是 HashMap 在处理哈希冲突时的解决办法；

    其实就是将数组和链表结合起来，发挥两者优势；

  • 再哈希法

    当发生冲突后，使用另外一个哈希函数获取地址，直到找到合适的地址

  • 建立公共溢出区

    将哈希表分为基本表和溢出区，冲突的元素放溢出区

  • 一致性哈希

    主要用于分布式系统中，如：分布式缓存，是将数据均分到不同的节点上以减少冲突

零碎的点

1：值传递和引用传递

除了集合、数组、对象会改变最终的值，因为三者是引用传递，其余都不会改变，因为其余的都是值传递，你传的是值过去，里面

做啥操作，跟外面没关系；

2：& 和 &&

&& 有短路功能；

&& 的左边是fasle, 右边就不再执行了；

3：抽象类和接口的区别

1：接口

  • 接口的方法默认是 public 修饰；
  • 接口不能有构造方法；
  • 从设计层面来说：接口相当于一种规范；

2：抽象类

  • 一个类只能有一个父类；

  • 从设计层面来说：抽象类相当于一种模板设计；

  • 抽象类可以有构造方法；

  • 抽象类主要是为了复用；我们经常说的模板方法模式；

4 ：成员变量和局部变量的区别

成员变量

​ 属于类变量或者实例；

​ 如果是 sttaic 修饰就是类变量，如果没有static 修饰就是实例变量；

​ 成员变量随着类的创建而存在；

​ 成员变量会自动赋值默认值；

局部变量

​ 属于方法中定义的变量或者方法参数；对象存在堆内存，局部变量也存在栈内存

局部变量随着方法的调用结束而结束；

​ 局部变量不会自动赋值；

5 ： 静态方法和实例方法

静态只能访问静态的，不能访问实例方法；

实例方法没有这方面的限制；

6：final 关键字

能作用的地方

  • 类
  • 方法
  • 变量

7 ：深拷贝和浅拷贝

深拷贝是创建一个一模一样的新对象，新旧对象之间不共享任何内存，新对象的修改不会影响老对象；

浅拷贝，拷贝的是地址；

无论是深拷贝还是浅拷贝，都需要实现 cloneable 接口；并在 clone 方法中重写拷贝的逻辑

8：如何理解多态

多态的概念比较简单：简单来说就是同一个操作作用于不同的对象，可以有不同的解释，产生不同的结果；

如果按照这个理解，多态是运行期的一种状态， Java 的重写是多态的体现，不过也有人提出重写是静态多态的提现，这个问题在 StackOverflow 上有很多讨论，但我更倾向于动态多态；

要实现这个运行期的动态绑定，就要满足 3 个条件：

​ 1：有类的继承或者实现

​ 2：要有重写

​ 3：父类引用指向子类对象

9：重载和重写的区别

1：重写是运行期的概念，只是参数列表不同；重载是编译期的概念，发生在父子类之间

2：运行期绑定，就是说在运行的时候，根据变量的引用指向子类对象，来调用具体的方法；

​ 编译期绑定，就是说在编译期根据参数的不同来决定调用哪个方法；

6： 快速失败机制

集合中有使用到快速失败机制；

在集合中为了避免并发修改异常，会维护一个 expectedModCount 期望修改次数的字段，表示集合被迭代器预期修改的次数。

还有一个 modeCount 字段，表示实际被修改的次数。在集合被修改的过程中，会去比较这俩值，如果不一致，就会触发快速失败机制。抛出 ConcurrentModifyException，并发修改异常。

当多个线程对结合进行修改时候，会抛出这个异常。

在 增强 for 循环中，对集合中元素增加删除操作，会抛出这个异常；

1：报错原因解析

modeCount 是集合的成员变量，而 expextedModeCount 是Iterator 迭代器内部定义的变量；

在遍历过程中，对结合的增加和删除，只会修改 modeCount 的值，而不会修改 expectedModeCount的值，导致这俩值

不相等。

1：ConpyOnWrite 的 add/remove 方法都加了锁，为啥还要copy 一份

CopyOnWrite 当只有要 add 或者 remove 的时候，才 Copy 一份去修改，这是一种延时懒惰策略；

它是一种写时复制的容器；

修改完毕，再将原来容器指向 Copy 的容器地址；

它的 add 和 remove 方法加锁，是为了防止在 Copy 的时候，Copy 出来多个副本，导致并发写；

但是它的读操作没有加锁；

这样做的好处是，我们可以并发读，当然这里读取到的数据有可能不是最新的，

因为写时复制的思想是通过延时懒惰更新的策略，来实现数据最终一致性，并非强一致性。

因此，CopyOnWrite 是一种读写分离的思想，读和写不同的容器, 使用于都多写少的场景；

常用于：白名单黑名单，商品类目的访问或者更新场景。

1：枚举的特点

枚举其实就是一个语法糖，定义的 Enum 类型，反映到字节码文件中就是 final class XXX extends java.lang.Enum

定义的属性反映到字节码上是：public static final aa

  • 构造方法私有，保证了只读不写，针对只读不写，我们只提供 get 方法，不提供 set 方法；
  • 该类是 final 修饰、hashCode 方法也是 final 修饰、属性是 static final 修饰
  • 枚举其实继承了 Enum 类
