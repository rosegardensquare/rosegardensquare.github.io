Dubbo 基本知识

1：Dubbo 基本知识

  • 传输协议

    默认 Dubbo 协议，但是也支持 Http 协议和其他协议

  • 通信框架

netty nio 异步通信框架

  • 序列化协议

默认 hession 协议，还有 Fastjason2

  • 传输体积

    8M

  • 长连接

2：Dubbo 缓存机制

就是服务端和客户端分别缓存调用结果，是为了提高查询效率；

客户端缓存起来是为了下次不再远程调用，服务端缓存起来是下次不再调用具体方法；

缓存淘汰策略有：

  • LRU

缓存机制有可能造成数据不一致问题，分布式缓存没事。

3：Dubbo 如何实现像本地调用一样调用全程服务

Dubbo 能实现本地方法一样调用远程方法的是基于动态代理。

Dubbo 根据动态代理生成一个代理类，这个代理实现了本地接口，在调用本地方法时候，会通过代理类的 invoke 方法转发到远程接口的方法

4：Dubbo 整体架构

Dubbo 的架构主要有：

  • service 层

    是一个扩展层

  • 配置层

  • 注册中心层

  • 代理层

  • 监控层

  • 网络传输层

  • 通信层

  • 序列化层

  • 集群层

    负载均衡、集群容错

  • 远程调用层

5：Dubbo 的工作原理

也就是原理和实现过程

具体的是：

  • 调用方

    • 配置层

      调用者在启动的时候读取配置层的配置；

    • 注册中心层

      根据上一步配置要找的接口，从注册中心层拉取和订阅对应的服务列表，缓存到本地，这样即使注册中心挂了，也能继续工作；

    • 代理层

      调用者在代理层生成调用者的代理对象 Invoker;

    • 集群层

      经过集群层负载均衡找到提供者的 Invoker

    • 通信层

      通过 Netty 框架通信

    • 序列化层

      序列化参数

  • 提供者

    • 序列化层

      反序列化参数

    • 然后交给线程池来执行

6：cluster 集群层

1：Dubbo 的负载均衡策略

它是在客户端设置的负载均衡。

  • 默认是加权随机；加权随机就是：通过区间的随机算法，去获取一个目标服务器。在这个区间中，我们可以对某一个服务器加权重。

  • 加权轮询

  • 最小活跃优先

    每一个服务者都记录一个调用的 active 数，完成调用之后，活跃数会减1。活跃数下降的越快，这台服务器的性能越高。

  • 最短时间优先

    他会去计算服务器的响应时间，根据响应时间最短的服务器配置更高的权重。

  • 一致性 Hash

    通过一致性哈希算法，会落到某一个目标服务器上。

2：集群容错重试

  • 默认是：失败切换到其他机器

    可以配置重试次数，默认重试2次；

  • 忽略异常

    对于不重要的业务，如：日志

  • 并行调用

    并行调用多个提供者，只要有一个成功，就返回；

7：Proxy 代理层

默认使用字节码动态代理

8：Dubbo 的扩展点 SPI

1：啥是 SPI

JDK 的 SPI 是一种对服务的可扩展的机制，它是基于接口的扩展。

就是我有一个接口，现在这个接口有 3 个实现类，那么系统在运行的时候，到底要加载哪一个实现类；这时候就需要 SPI；

根据指定的配置或者默认配置，去找到对应的实现类加载进来；

具体使用步骤：

第一步：在配置文件中写

第二步：

代码中使用 ServiceLoader.load 加载实现类，通过反射创建实例执行方法。

2：SPI 的使用场景

扩展某个功能

  • JDBC 加载不同类型的数据库驱动

    java 定义了一套 jdbc 的接口，但是没有实现类；实际项目中，它加载的是你配置文件中指定的 jdbc 信息

    它是在类加载的时候就加载所有的扩展，不是延迟加载。

3：Dubbo 的 SPI

Dubbo 的 SPI ，它是在 java spi 的基础上增加了新的功能：按需加载、懒加载；

它是放在Mata-info/dubbo 文件下；

它是用扩展类加载器加载；

他能扩展许多功能：

  • 传输协议
  • 序列化协议
  • 负载均衡

主要作用就是对外提供可自定义的插件式的扩展 Dubbo 功能。

9：Dubbo 是如何动态感知服务下线的呢

首先 Dubbo 默认采用 ZK 作为注册中心。

多个服务之间的通信地址通过 ZK 类维护，zk 会采用树型结构存储协议地址；ZK 通过心跳机制，判断提供者是否健康，然后决定把哪个服务剔除。zk 是咋感知的呢

简单来说就是 ZK 的 watch 监听机制，监听服务端节点的变化，一旦有变化，就发送时间通知消费者做出更新本地服务列表。后续就不会发送到有问题的节点上了。
