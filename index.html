Spring IOC

1：啥是 Spring IOC

所谓的 IOC 意思是控制反转；

啥是控制反转呢？

相对于控制反转，传统的方式想使用一个对象，就是通过 new 的方式创建。

而 IOC，它是 spring 容器去创建对象并放在容器中，当需要的时候直接从容器中拿就行了。这个创建对象的控制权交给了spring。

IOC 是一种指导思想。它有许多实现方式.

2: IOC 咋落地实现的呢

spring 从元数据拿到需要依赖注入的业务 POJO。

这个元数据可以是：xml、configuration 注解中的 bean 注解、各种bean注解。

拿到 POJO 之后，生成 BeanDefition，然后注入到spring 容器中。用的时候，要经过spring 上下文获取容器中的对象。

Spring bean 的生命周期

总的来说分为但三大步：创建、使用、销毁

细分：

0.1：单例、多例检查

在实例化之前，会检查 bean 的作用域，如果是单例的，就在容器中找，有没有，有的化就被重用。

没有的话，才走实例化。

1：实例化

​ spring 找到 bean 的元数据之后，首先会实例化 bean，如果有工厂方法，就用工厂方法创建，如果没有，就调用构造器创建。

​ 这个过程就相当于新生儿出生，还未觉醒。

1.1：实例化之后先使用三级缓存解决循环依赖

2：属性填充

​ spring 根据元数据信息，为 bean 的属性赋值；相当于给新生儿穿衣打扮

3：Aware 接口注入

​ 如何 bean 实现了 aware 接口，此时会执行那么方法。让 bean 知道自己的名字、所在的上下文；

​ 相当于新生儿意识到自己是谁，在哪里

4：初始化前置

​ BeanPostProcessor 的前置方法，可以改变当前状态的 bean ；

​ spring 遍历所有BeanPostProcessor 的实现类，执行前置方法；

​ 相当于给孩子做一些额外工作；

​

4：初始化

​ 真正的初始化。主要是 InitBean 接口。@PostConstruct 注解就是在这个阶段。

​ 按顺序执行：@PostConstruct、afterPropertiesSet、 init-method、

​ 相当于孩子开始上学，接受教育。

5：初始化后置

​ BeanPostProcessor 的后置方法，对于初始化完成的bean 再次加工。

​ spring 遍历所有BeanPostProcessor 的实现类，执行后置方法；

​ 相当于对孩子不断完善。

5.1：aop 阶段

BeanPostProcessor 的两个后置方法专门用来处理 aop：xxAspectJxx.

6：销毁前置

​ DisPonseAble 接口指定的销毁方法

7：销毁

​ Bean 被销毁，释放资源，生命周期结束。

AOP

1：AOP 是啥

1：AOP 的概念

AOP 是面向切面编程；

就是把公共的逻辑抽出来，开发人员只用关注业务就行。

aop 是一种指导思想，它对接的是切面，具体实现有几种方式。

2：AOP 中的几个点

  • 切面 = 切入点 + 通知

    就是定义了干什么和什么时候干的逻辑，使用 @PointCut 注解

  • 切入点

    表示应该拦截哪些方法，使用 execution 表达式

  • 通知

    表示拦截之后要做什么事情；

    • 前置通知
    • 后置通知
    • 前置后置通知
    • 异常后的通知
    • finally 最终通知

  • 连接点

  • 目标对象

3：实际工作中的作用场景

  • 减少代码耦合、提高复用；

  • 对方法增强；

比如：想要监控一个方法的整体耗时；

于是定义一个切面，然后使用环绕通知，@around 注解；在方法执行前后加入监控的逻辑；

后期可以改进使用普罗米修斯 + granfna 来监控；

2：AOP 咋思想落地的呢

AOP 要融入spring bean 生命周期的初始化过程，主要是 BeanPostProcessor 这个对象的两个方法起作用。

  • 初始化后置

    spring 会判断是否需要为当前bean 创建代理，主要还是 beanpostprocessor 的两个方法起作用，它们会根据切面信息，判断是否需要创建代理对象。

    • 如果要创建代理对象

      BeanPostProcessor 会用代理工厂或者 JDK 动态代理或者 CGLIB 代理创建代理对象，代理对象会在配置的通知点织入。

    • 代理创建后

      BeanPostProcessor 将原始对象替换为代理对象。从而保证后续调用这个bean 都是走的代理对象。

3：AOP 的两种实现方式

基于动态代理模式实现；

  • 默认是 JDK 动态代理

    使用反射，在运行期动态创建目标接口的实现类，在执行方法的时候走的是代理类 invoke 方法，然后它内部调用原始类的方法，在方法前后增强。

    只能基于接口代理；

  • CGLIB 动态代理

​ 它是在运行时动态生成目标类的子类，不受接口的限制，它是基于字节码的操作，性能可能要好一点。

RPC

1：啥是 RPC

RPC 是远程过程调用，是一种进程间的通信方式。

它关注几个要素：

  • 传输协议、

  • 通信框架

  • 序列化协议、

  • 集群、

    负载均衡、集群容错

  • 服务注册与发现就是服务寻址

  • 动态代理方案实现像本地调用一样调远程服务

  • 查询结果缓存起来

HTTP 是超文本传输协议，它只是一个协议。

RPC 通常用于服务间的调用，HTTP 常用于 web、app、第三方的调用；

HTTP 的实现是基于 TCP ，RPC 也可以基于 TCP，或者 HTTP 实现。

2：RPC 框架有哪些

  • Dubbo
  • 阿里的 HSF
  • 谷歌的 grpc

3：为啥 RPC 比 HTTP 更快一些

RPC 的设计目的就是为了快速传输，因此它的序列化比 HTTP 的更紧凑，没有像 HTTP 传输的还要带上 Header、body 那样，RPC 相对传输的数据量更少。

3：RPC 原理，如何设计一个 RPC 框架

首先一个 RPC 框架的几个要素是：

  • 配置层

    可以 SPI 扩展

  • 注册中心层

  • 代理层

  • 集群层

  • 监控层

  • 网络传输

  • 序列化

  • 通信协议

通信框架可以使用高性能的 netty，不用自己造轮子；

要实现像本地方法一样调用远程方法，还得屏蔽上述细节，那就要使用代理来实现。要做到高可用，就得使用集群、还有服务注册中心、还得有一套完善的监控机制。

1：ZK 的应用场景引出 zk 的两个核心：

1：全局唯一

3：选举机制

1：选举机制原理

ZK 的选举发生在节点刚启动或者Leader崩溃时候；

选举的目的是选出一个 Leader;

节点状态有：

  • Looking 竞选状态
  • Leader
  • Follower
  • Observer 观察者

选举的本质是进行 PK，PK 的依据是节点的能力，谁的能力越强就胜出。每个节点都有事务id, 节点编号 myid，事务id 越大代表数据最新，能力越强。先比较事务id, 相同的话再比较 myid。

投票的过程中，每个节点内部维护一个投票结果，类似于一个投票箱，开始投票时，每个节点先投自己一票，然后互相交换选票，根据 PK 规则，更新自己维护的选票结果，之后再广播出去，统计每个节点的票数，超过节点总数一半的节点胜出。

2：选举机制引入脑裂问题

  • 啥是脑裂

​ 所谓脑裂就是，当集群中出现网络通信故障时候，各种原因导致 Leader 跟 Foller 节点通信异常，Foller 节点认为Ldear 挂掉，于是 重新选举Leader, 当网络恢复后，集群中出现两个 Leader, 客户端在连接的时候就会不知道连接哪个Leader。

  • 如何解决脑裂

​ 采用过半机制 + zxid 中的任期。

3：zk 为啥推荐奇数节点为防止脑裂和容错

集群能正常，只要有半数以上存活就行。

  • 一共有 2 个节点，一半是 1，半数以上是 2 ，允许宕机是 0 ；
  • 一共有 3 个节点，一半是 1，半数以上是 2 ，允许宕机是 1 ；
  • 一共有 4 个节点，一半是 2，半数以上是 3 ，允许宕机是 1 ；
  • 一共有 5 个节点，一半是 2，半数以上是 3 ，允许宕机是 2 ；
  • 一共有 6 个节点，一半是 3，半数以上是 4 ，允许宕机是 2 ；

总结就是：

​ 基数台和偶数台，允许宕机的数量是相同的。就不用浪费多部署一台了。

为了两个方面的考虑：

  • 集群容错

  • 防止脑裂

    对于双机房来说：

    ​ 如果是 6 台，每个机房都是3台，不考虑过半机制，他们都能选出主节点，考虑的话都选不出；

    ​ 如果是 5 台，一个机房是 3 台，一个是 2 台，考虑和不考虑，其中有一边都能选出。

    因此，最好是奇数。

Dubbo 基本知识

1：Dubbo 基本知识

  • 传输协议

    默认 Dubbo 协议，但是也支持 Http 协议和其他协议

  • 通信框架

netty nio 异步通信框架

  • 序列化协议

默认 hession 协议，还有 Fastjason2

  • 传输体积

    8M

  • 长连接

2：Dubbo 缓存机制

就是服务端和客户端分别缓存调用结果，是为了提高查询效率；

客户端缓存起来是为了下次不再远程调用，服务端缓存起来是下次不再调用具体方法；

缓存淘汰策略有：

  • LRU

缓存机制有可能造成数据不一致问题，分布式缓存没事。

3：Dubbo 如何实现像本地调用一样调用全程服务

Dubbo 能实现本地方法一样调用远程方法的是基于动态代理。

Dubbo 根据动态代理生成一个代理类，这个代理实现了本地接口，在调用本地方法时候，会通过代理类的 invoke 方法转发到远程接口的方法

4：Dubbo 整体架构

Dubbo 的架构主要有：

  • service 层

    是一个扩展层

  • 配置层

  • 注册中心层

  • 代理层

  • 监控层

  • 网络传输层

  • 通信层

  • 序列化层

  • 集群层

    负载均衡、集群容错

  • 远程调用层

5：Dubbo 的工作原理

也就是原理和实现过程

具体的是：

  • 调用方

    • 配置层

      调用者在启动的时候读取配置层的配置；

    • 注册中心层

      根据上一步配置要找的接口，从注册中心层拉取和订阅对应的服务列表，缓存到本地，这样即使注册中心挂了，也能继续工作；

    • 代理层

      调用者在代理层生成调用者的代理对象 Invoker;

    • 集群层

      经过集群层负载均衡找到提供者的 Invoker

    • 通信层

      通过 Netty 框架通信

    • 序列化层

      序列化参数

  • 提供者

    • 序列化层

      反序列化参数

    • 然后交给线程池来执行

6：cluster 集群层

1：Dubbo 的负载均衡策略

它是在客户端设置的负载均衡。

  • 默认是加权随机；加权随机就是：通过区间的随机算法，去获取一个目标服务器。在这个区间中，我们可以对某一个服务器加权重。

  • 加权轮询

  • 最小活跃优先

    每一个服务者都记录一个调用的 active 数，完成调用之后，活跃数会减1。活跃数下降的越快，这台服务器的性能越高。

  • 最短时间优先

    他会去计算服务器的响应时间，根据响应时间最短的服务器配置更高的权重。

  • 一致性 Hash

    通过一致性哈希算法，会落到某一个目标服务器上。

2：集群容错重试

  • 默认是：失败切换到其他机器

    可以配置重试次数，默认重试2次；

  • 忽略异常

    对于不重要的业务，如：日志

  • 并行调用

    并行调用多个提供者，只要有一个成功，就返回；

7：Proxy 代理层

默认使用字节码动态代理

Java 泛型

1：对泛型的理解

泛型是 jdk 1.5 开始的，允许在定义类、接口、方法的时候使用类型参数；定义的类型参数在使用的时候会用具体的类型替换。泛型最主要是用在集合中或者封装的一些组件中；

简单来说，Java 泛型就是在创建或者调用的时候才明确下具体的类型；

2： 泛型的好处

1：方便

提高代码的复用性；假如有一个 List，不用泛型，想存字符串得 new 一个，想存数字，又得 new 一个。此时，可以使用泛型List

2：安全

没有泛型的时候，通过向上转型存入任何类型的元素；在获取的时候就麻烦了，向下转型可能会出现类型转换异常。

泛型是在编译期间做检查，不用再等到运行期间报 classcastException 异常；

泛型其实是一种代码模版，可以理解为广泛的类型。

3：泛型是如何实现的

通过类型擦除实现。说白了就是一种语法糖形式。在编译期将 List 转化为 List

1 : 关于类型擦除

泛型的信息只存在于编译期，在编译阶段会将所有的 T 替换为 Object , 这个叫做类型擦除。

可以理解为：在编译期间，会拿着<> 中的类型去检查类型是否匹配，检查通过，擦除并替换为 Object。

2 ： 啥是类型擦除

类型擦除是 Java 处理泛型的一种方式，写的代码是 List、List, 编译后的字节码文件都会变成 List，

把泛型信息都擦除了，这个过程叫做类型擦除；

虽然编译后的字节码文件中不会再有泛型的信息，但是运行期，程序还是可以通过：插入类型转换和桥方法保证类型安全；

3：泛型被擦除了，如何知道真实类型

泛型的信息只存在于编译期，生成的 class 文件中被擦除；

但是泛型的擦除是有范围的，定义在类上的泛型信息不会被擦除；

Java 编译器仍在 class 文件中以 singnature 属性的方式，保留泛型的信息；

Type 作为顶级接口，下面有几种类型：

​ TypeVariable;

​ paramterTyep;

​ wildcardType;

​ GenericArrayType;

通过这些接口，我们就可以在运行期获取泛型信息；

3：为啥要有类型擦除

1：是为了保持与之前的没有泛型的类型的兼容，起到向前 jdk 版本兼容；

2：在编译期就可以进行类型判断，而不用在运行期判断，提高性能；

4：泛型在集合中使用

泛型在集合中使用的好处是：限定了存入的元素类型，避免类型错误，存取都不会出现类型转换异常；

如果你要在集合中使用 Object 类型，存入的时候，你可能会存入不同的类型，取的时候强转可能会转换异常。

所以 Java 中引入泛型的概念，也是对集合中的使用更加标准化。

5 ：泛型 extends、super

泛型的 extends、super （协变、逆变）

其实解决的就是一个参数范围：向上或者向下

1：协变

2：逆变

3 ： extends 和 super 什么时候用？

啥时候用，其实这里要遵守一个 PECS 原则；

就是说，泛型的所在的对象，它在未来使用的时候，它是生成者还是消费者，如果是生产者那就用 extends;

如何集合是一个消费者，对外产生元素供读取，那么此时使用 extends;

举个例子：

​ 如果你要从一个泛型集合中读取一个元素，那么使用 extends, extends 只能读取元素不能添加元素;

​ 如何是往泛型集合中添加元素，那么使用 super，super 只能添加元素，不能读取元素;

  • static 修饰的变量叫做类变量或者静态变量，在内存中只有一份；它跟 new 无关，所有的实例共享一份

      随着类的加载而加载；因为类只加载一次，因此静态也只加载一次

  • 普通的变量叫做实例变量，存在堆中，每个对象有各自的；

    • 加载时机：

      随着对象的创建而创建

静态下不能使用 this 或者 super;

普通可以调任何；

原子类

1：原子类相关

对于 i++操作，它其实是分为三步骤，无法使用 volatile 保证原子性。可以使用sync 锁。

或者原子类，因此，引入原子类。

原子类底层是：

cas + volatile

原子类内部调用 unsafe 类的 cas 方法。

2：unsafe

它是CAS 的核心类；

因为 Java 无法直接访问底层操作系统，而是通过本地native方法访问；

但是 JVM 留了一个后门，那就是 java 中的 unsafe 类。

它提供了硬件层面的原子操作。

unsafe 类提供了需要基础方法：

  • 数组操作

  • 对象操作

  • CAS

  • 分配对象的内存空间，反射创建对象

  • 线程的 park、unpark 方法

3：CAS

1：啥是 CAS

CAS 是乐观锁的一种实现。

是 compare and swap 的简称；

意思是，先比较再替换。

它有三个参数值：

  • V：内存地址
  • E：期望这个内存地址当前值是啥
  • B：想要更新成啥

在并发修改时，先取出内存地址中的值，对比是不是期望的值，如果是的话，就更新成新值。如果不是，其他线程也不会阻塞和挂起，而是再次尝试。

CAS 保证同一个时刻只能有一个线程更新成功。

2：CAS 为啥能保证只有一个线程可以成功呢

CAS 是一种原子操作，在操作系统层面，它的原理是基于硬件提供的原子指令。

在 X86 架构的CPU中，它底层依赖 cmpxchg 指令。

1： cmpchg 指令的作用

1：保证原子性

它是一个原子指令，CPU 在运行这个指令的时候，它会锁总线，防止其他CPU访问；

而且它在操作期间，防止CPU中断，保证一组操作肯定能执行完毕。

3：CAS 对比 Sync 锁

相对于 Sycn 锁的阻塞算法，CAS 是非阻塞算法；

CAS 主要用于实现：

  • 乐观锁
  • 自旋锁

4：CAS 出现 ABA 问题

1：ABA 问题是啥

CAS 的 ABA 问题是指在使用 CAS 时，由于CAS 算法实现的一个重要前提是：取出内存中某时刻的值，在下一个时刻比较并且交换，那么在这个时间差内，有可能数据被篡改。

比如：有两个线程 t1,t2;

都从相同内存位置取出值 A，线程1修改为B，此时线程2以为某种原因让出CPU时间挂起了，线程1再把B改为A，之后呢，线程2恢复执行，比较内存地址是A，认为没有被改过，就继续执行自己的操作。线程2 的操作过程违背了 CAS 的思想。这就是 ABA 问题。

2：如何解决 ABA 问题

有两种方案：

  • 加版本号

    对内存位置的值，没修改一次，就对版本号 +1；

    在比较内存位置的值时候，也要比较下版本号有没有变化过，如果两个都没变过，那就执行成功。

  • Java 中提供的 AtomicStampedReference

    它维护了一个引用和一个时间戳。

    它检查引用地址是否被改过的同时，也会检查时间戳有没有被改过。

5：乐观锁和悲观锁

1：乐观锁

乐观锁是一种思想。它假设数据一般情况下不会造成冲突，所以在数据提交更新的时候，才会检查数据是否冲突。

乐观锁的实现细节：

  • 检查冲突
  • 数据更新

最典型的就是 CAS

2：悲观锁

悲观锁认为：数据被并发修改的概率比较大。在修改之前先要加上锁。防止被篡改。

Sync 锁就是一种悲观锁。
