RPC

1：啥是 RPC

RPC 是远程过程调用，是一种进程间的通信方式。

它关注几个要素：

  • 传输协议、

  • 通信框架

  • 序列化协议、

  • 集群、

    负载均衡、集群容错

  • 服务注册与发现就是服务寻址

  • 动态代理方案实现像本地调用一样调远程服务

  • 查询结果缓存起来

HTTP 是超文本传输协议，它只是一个协议。

RPC 通常用于服务间的调用，HTTP 常用于 web、app、第三方的调用；

HTTP 的实现是基于 TCP ，RPC 也可以基于 TCP，或者 HTTP 实现。

2：RPC 框架有哪些

  • Dubbo
  • 阿里的 HSF
  • 谷歌的 grpc

3：为啥 RPC 比 HTTP 更快一些

RPC 的设计目的就是为了快速传输，因此它的序列化比 HTTP 的更紧凑，没有像 HTTP 传输的还要带上 Header、body 那样，RPC 相对传输的数据量更少。

3：RPC 原理，如何设计一个 RPC 框架

首先一个 RPC 框架的几个要素是：

  • 配置层

    可以 SPI 扩展

  • 注册中心层

  • 代理层

  • 集群层

  • 监控层

  • 网络传输

  • 序列化

  • 通信协议

通信框架可以使用高性能的 netty，不用自己造轮子；

要实现像本地方法一样调用远程方法，还得屏蔽上述细节，那就要使用代理来实现。要做到高可用，就得使用集群、还有服务注册中心、还得有一套完善的监控机制。

  • 加权轮询

  • 最小活跃优先

    每一个服务者都记录一个调用的 active 数，完成调用之后，活跃数会减1。活跃数下降的越快，这台服务器的性能越高。

Java 泛型

1：对泛型的理解

泛型是 jdk 1.5 开始的，允许在定义类、接口、方法的时候使用类型参数；定义的类型参数在使用的时候会用具体的类型替换。泛型最主要是用在集合中或者封装的一些组件中；

简单来说，Java 泛型就是在创建或者调用的时候才明确下具体的类型；

2： 泛型的好处

1：方便

提高代码的复用性；假如有一个 List，不用泛型，想存字符串得 new 一个，想存数字，又得 new 一个。此时，可以使用泛型List

2：安全

没有泛型的时候，通过向上转型存入任何类型的元素；在获取的时候就麻烦了，向下转型可能会出现类型转换异常。

泛型是在编译期间做检查，不用再等到运行期间报 classcastException 异常；

泛型其实是一种代码模版，可以理解为广泛的类型。

3：泛型是如何实现的

通过类型擦除实现。说白了就是一种语法糖形式。在编译期将 List 转化为 List

1 : 关于类型擦除

泛型的信息只存在于编译期，在编译阶段会将所有的 T 替换为 Object , 这个叫做类型擦除。

可以理解为：在编译期间，会拿着<> 中的类型去检查类型是否匹配，检查通过，擦除并替换为 Object。

2 ： 啥是类型擦除

类型擦除是 Java 处理泛型的一种方式，写的代码是 List、List, 编译后的字节码文件都会变成 List，

把泛型信息都擦除了，这个过程叫做类型擦除；

虽然编译后的字节码文件中不会再有泛型的信息，但是运行期，程序还是可以通过：插入类型转换和桥方法保证类型安全；

3：泛型被擦除了，如何知道真实类型

泛型的信息只存在于编译期，生成的 class 文件中被擦除；

但是泛型的擦除是有范围的，定义在类上的泛型信息不会被擦除；

Java 编译器仍在 class 文件中以 singnature 属性的方式，保留泛型的信息；

Type 作为顶级接口，下面有几种类型：

​ TypeVariable;

​ paramterTyep;

​ wildcardType;

​ GenericArrayType;

通过这些接口，我们就可以在运行期获取泛型信息；

3：为啥要有类型擦除

1：是为了保持与之前的没有泛型的类型的兼容，起到向前 jdk 版本兼容；

2：在编译期就可以进行类型判断，而不用在运行期判断，提高性能；

4：泛型在集合中使用

泛型在集合中使用的好处是：限定了存入的元素类型，避免类型错误，存取都不会出现类型转换异常；

如果你要在集合中使用 Object 类型，存入的时候，你可能会存入不同的类型，取的时候强转可能会转换异常。

所以 Java 中引入泛型的概念，也是对集合中的使用更加标准化。

5 ：泛型 extends、super

泛型的 extends、super （协变、逆变）

其实解决的就是一个参数范围：向上或者向下

1：协变

2：逆变

3 ： extends 和 super 什么时候用？

啥时候用，其实这里要遵守一个 PECS 原则；

就是说，泛型的所在的对象，它在未来使用的时候，它是生成者还是消费者，如果是生产者那就用 extends;

如何集合是一个消费者，对外产生元素供读取，那么此时使用 extends;

举个例子：

​ 如果你要从一个泛型集合中读取一个元素，那么使用 extends, extends 只能读取元素不能添加元素;

​ 如何是往泛型集合中添加元素，那么使用 super，super 只能添加元素，不能读取元素;

原子类

1：原子类相关

对于 i++操作，它其实是分为三步骤，无法使用 volatile 保证原子性。可以使用sync 锁。

或者原子类，因此，引入原子类。

原子类底层是：

cas + volatile

原子类内部调用 unsafe 类的 cas 方法。

2：unsafe

它是CAS 的核心类；

因为 Java 无法直接访问底层操作系统，而是通过本地native方法访问；

但是 JVM 留了一个后门，那就是 java 中的 unsafe 类。

它提供了硬件层面的原子操作。

unsafe 类提供了需要基础方法：

  • 数组操作

  • 对象操作

  • CAS

  • 分配对象的内存空间，反射创建对象

  • 线程的 park、unpark 方法

3：CAS

1：啥是 CAS

CAS 是乐观锁的一种实现。

是 compare and swap 的简称；

意思是，先比较再替换。

它有三个参数值：

  • V：内存地址
  • E：期望这个内存地址当前值是啥
  • B：想要更新成啥

在并发修改时，先取出内存地址中的值，对比是不是期望的值，如果是的话，就更新成新值。如果不是，其他线程也不会阻塞和挂起，而是再次尝试。

CAS 保证同一个时刻只能有一个线程更新成功。

2：CAS 为啥能保证只有一个线程可以成功呢

CAS 是一种原子操作，在操作系统层面，它的原理是基于硬件提供的原子指令。

在 X86 架构的CPU中，它底层依赖 cmpxchg 指令。

1： cmpchg 指令的作用

1：保证原子性

它是一个原子指令，CPU 在运行这个指令的时候，它会锁总线，防止其他CPU访问；

而且它在操作期间，防止CPU中断，保证一组操作肯定能执行完毕。

3：CAS 对比 Sync 锁

相对于 Sycn 锁的阻塞算法，CAS 是非阻塞算法；

CAS 主要用于实现：

  • 乐观锁
  • 自旋锁

4：CAS 出现 ABA 问题

1：ABA 问题是啥

CAS 的 ABA 问题是指在使用 CAS 时，由于CAS 算法实现的一个重要前提是：取出内存中某时刻的值，在下一个时刻比较并且交换，那么在这个时间差内，有可能数据被篡改。

比如：有两个线程 t1,t2;

都从相同内存位置取出值 A，线程1修改为B，此时线程2以为某种原因让出CPU时间挂起了，线程1再把B改为A，之后呢，线程2恢复执行，比较内存地址是A，认为没有被改过，就继续执行自己的操作。线程2 的操作过程违背了 CAS 的思想。这就是 ABA 问题。

2：如何解决 ABA 问题

有两种方案：

  • 加版本号

    对内存位置的值，没修改一次，就对版本号 +1；

    在比较内存位置的值时候，也要比较下版本号有没有变化过，如果两个都没变过，那就执行成功。

  • Java 中提供的 AtomicStampedReference

    它维护了一个引用和一个时间戳。

    它检查引用地址是否被改过的同时，也会检查时间戳有没有被改过。

5：乐观锁和悲观锁

1：乐观锁

乐观锁是一种思想。它假设数据一般情况下不会造成冲突，所以在数据提交更新的时候，才会检查数据是否冲突。

乐观锁的实现细节：

  • 检查冲突
  • 数据更新

最典型的就是 CAS

2：悲观锁

悲观锁认为：数据被并发修改的概率比较大。在修改之前先要加上锁。防止被篡改。

Sync 锁就是一种悲观锁。

lock 锁

lock 接口下还有很多实现类。。。

  • ReadWriteLock 读写锁

  • Condition 接口，实现等待通知模式

  • StampedLock 邮戳锁、票据锁，它是优化读写锁的缺点才引入

1：ReentrantLock 用法

由于 Sync 锁很重，JUC lock 包中，引入 ReentrantLock 锁，替代 Sync 锁；

ReentrantLock 内部基于 AQS 实现；

它和 Sync 锁一样，也是一个可重入锁；

2：ReentrantLock 如何获得可重入

它在加锁的的时候，先看下当前获得到锁的线程是不是自己，如果是的话，就可以再次获取到这个锁，只需要简单的把 state 加就行，增加可重入的次数。

当它要释放锁的时候，那就得确保 state = 0 才能释放，加锁几次，解锁也几次。

3：ReentrantLock 的公平和非公平性

默认是非公平，可以通过它的构造方法设置。

1：非公平锁

多个线程不按照申请锁的顺序去获取锁；而是直接尝试 CAS 获取锁，要是获取不到，再进入等待队列不迟；

获取不到锁，就把当前线封装成一个 node 放到队列尾部。这个队列是一个双向队列。

1：代码层面

非公平锁的核心在内部类 NonFairSync ，它实现了 AQS 中的 Sync 类。

2：公平锁

多个线程按照申请锁的顺序去获取锁， 所有的线程都在队列中排队，这样就保证队列中第一个线程先获取到锁。

它进来之后，先看前面有没有前序节点，没有的话，就获取到锁；有的话，就进队列。

1：代码层面

公平锁的核心在内部类 FairSync 中，它继承 AQS中的 Sync。

它跟非公平锁，只有一点差别，那就是在获取锁之前，先判断下自己前面有没有人排队。

3-1：公平锁和非公平锁对比

两种锁分别适用于不同的场景，存在着各自的优缺点。

对于公平锁来说：它优点是，所有的线程都能获取到资源，不会饿死在队列中。但是它会造成吞吐量下降。队列中，没有获取到锁的，会被阻塞，CPU 唤醒线程的开销是最大缺点。

对于非公平锁来说：它可以减少 CPU 唤醒锁的开销，整体吞吐量会高一点，但是会导致队列中一直获取不到锁的线程长时间等待，造成锁饥饿。

4：ReentrantLock 中的 CAS 涉及

  • cas 在修改 volatile 修饰的 state 的时候，成功就代表获取锁
  • 入队列的操作也用到 CAS 。

    每一个对象都有一个计数器，用来记录被锁了几次，初始化值是0，每加一次锁，这个次数就加1。释放锁的时候，这个次数会减去1，直到0的时候，锁就被释放，其他线程后才能去获取锁。

  • 方法级别的锁

    它是隐式的。同步方法的常量池中会有一个 ACC_SYNC 的标志，当一个线程要访问同步方法时，会先去检查这个标志，如果有设置，需要先去获取监视器锁，方法执行完之后才释放锁。这时候其他线程进来，会因为没办法获取监视器锁而阻塞。如何执行方法的时候出现了异常，代码也没有捕获处理异常，那么异常在被抛出来之前监视器会自动把这个锁给释放了。

2：Syn 锁是可重入的

当一个线程获取到了锁，在没有释放锁之前还可以再次获取锁。

3：锁的对象不一样

  • 修饰方法
    • static 方法，那就是类锁
    • 普通方法，对象锁
  • 修饰代码块，括号里的内容
    • 锁的是 class ，就是类锁
    • 锁的是this, 那就是对象锁

4：对象的监视器 monitor

每个对象都有一个监视器。

当去获取锁的时候，就是对这个对象的监视器操作。

1：monitor 的数据结构

在 HotSpot 虚拟机中，monitor 是 ObjectMonmitor.java 对应 c++ 中的 ObjectMonitor.c++。底层是 ObjectMonitor 对象。

它有几个重要属性：

  • _owner：谁获取了 objectMonitor, 它就指向哪个线程
  • _EntrySet：等待获取锁的线程，这些线程都是阻塞状态。
  • _WaitSet：处理waiting 状态的线程
  • 重入锁的次数
  • 获取锁计数器

2：加锁的流程形象比喻

有一个大楼，有一个特殊房间，每次只能有一个人进入，所有的想进去的人，都得在走廊排队，这个走廊就是 entrySet；

有些特殊的已经获准进入的但是暂时无法脱身的人（比如线程挂起），它们会进入一个叫做 wiatSet 的房间，这些人稍后会被叫到重新进入特殊房间。

3：加锁在代码层面分析

当多个线程访问同步代码块的时候，会先进入 extrySet 队列，此时如果某一个线程活得了 objectMonitor 对象后：

  • 它首先会进入一个 Owner 的区域，把owner 属性设置为自己；
  • 设置计数器 + 1

这些弄完之后，就成功获取了锁。

如果这个线程调用了 wait 方法，它就会进入 waitSet 中，会把 owner 设置为 null，把计数器 -1。等待被唤醒。

4：sync 锁很重

jdk 1.6 之前加锁时， sync 会直接调用 ObjectMonitor 的enter 方法，这个操作是重量级的。为啥这么说呢？

因为 java 的自身不能启动线程，它的线程是基于操作系统深入到内核来完成的，应用程序创建线程后，调用 start 方法，此时操作系统就会从用户态切换到内核态，调用 c++ 源码中的 start0 方法，启动线程，因此这个操作很重。

5：锁优化

jdk 6 开始，sync 就做了优化：

  • 引入偏向锁、轻量级锁

  • 锁消除

    是 JVM 的 JIT 编译器对锁的优化，它是 JIT 编译期经过逃逸分析之后，发现加不加锁都不会发生线程安全。

    就是：经过分析，同步代码加不加锁都只会被同一个线程访问。那这时候就会锁消除。

    就比方说：我们经常定义局部变量 StringBuffer , 这时候 JIT 编译器就会帮锁消除。因为 StringBuffer 的方法加了 Sync 锁。

  • 默认开启自旋锁、自适应自旋

    自旋锁是 CPU 空转，没有放弃cpu 时间

5：Sync 能保证原子性、可见性、有序性

  • 保证原子性

    他能保证同一时间只有一个线程去操作共享变量

  • 可见性

    sync 锁在释放锁的时候，会将工作内存中的变量刷新到主内存中。这样，其他线程就能看到了。

  • 有序性

    共享变量只有在全部 sync 包裹中才能保证有序；如果在 sync 作用范围外使用了这些变量，那么它不会有序。

    如果想要严格的保证有序，得使用 volatile。

  • 对象的哈希码
  • 分代年龄
  • 锁状态

7：Sync 锁升级

  • 无锁

    当一个线程第一次访问同步代码时，JVM 会先在对象头设置当前访问的线程的id，并将状态设置为偏向锁。

  • 偏向锁

    偏向锁模式下，锁总是会偏向第一个获取锁的线程；

    此时，要是有其他线程过来，Jvm 会先检查对象头中偏向锁的标识，如果是自己的线程id，那就直接获取锁成功；如果

    不是自己的线程id：cas 修改偏向锁id，如果修改成功，当前线程获取了偏向锁；如果修改失败，

    就升级为轻量级锁，偏向锁撤销。偏向锁默认是关闭的。

    jdk 15 中偏向锁已经被废除。之前的版本中，jvm 对偏向锁是 4s 之后才启动。如果关闭偏向锁，锁从无锁直接到轻量级锁。

  • 轻量级锁

    有三个动作：

    • 复制 mark word

      JVM 将对象头中 mark word 复制到线程栈中的锁记录（lock record），这个过程叫做 displaced mark word。

      为啥要复制呢？

      ​ 原因是：为了保留对象的原始信息，为了在释放锁的时候能正确的恢复对象头的原始状态。因为锁的获取和释放成对的。

    • cas 更新 mark word

      尝试使用 cas 更新 mark word 为指向锁记录的指针；

      如果这个动作成功，那么这个线程就成功获取轻量级锁；

      如果这个动作失败，表示有其他线程以及获得到了轻量级锁；

    • cas 失败后自旋等待锁释放

      java 8 中轻量级锁的自旋自动开启，之前就引入了只是默认关闭的；

      此时锁会自旋等待锁释放；自旋 15 次之后（JDK 6 之前默认是10次），仍然失败，会升级为重量级锁。

      jdk 6 之后的版本开始，引入了轻量级锁的自适应自旋；

      ​ 它是根据前一次自旋的时间和成功获取锁的概率进行自适应调整。这些都是 C++ 写的。

  • 重量级锁

    升级为重量级锁之后，jvm 会在对象头中记录指向等待队列的指针。

    这种状态下，一个线程想要获取锁，得下进入等待队列阻塞，

    这里的获取重量级锁也有一个自旋：

    ​ jdk 8 中重量级锁自旋默认不开启。

    当锁释放之后，jvm 会从等待队列中选择一个线程唤醒。

    并将这个被选中的线程的状态设置为就绪状态，等待获取 cpu 时间片。

volatile

1：volatile 能保证原子性吗

不能，volatle 可以理解为轻量级锁sync 锁，但是它跟 sync 锁不同的是：它只能修饰共享变量。它只能保证可见性、有序性。

2：volatle 可见性

使用 volatile 修饰变量，写操作时，jvm 会向 cpu 发出 lock 前缀的指令，意思是将工作内存的数据更新到主内存。这样，其他 cpu 会遵守缓存一致性协议，将主内存的数据加载到自己的工作内存，那么其他线程就读到了最新的变量。

因此，它保证了可见性。

1：解决缓存一致性问题

由于 CPU 和主内存之间存在速度差异，为了弥补这个差异，人们在 CPU 和主内存之间加了几个高速缓存，有L1、L2甚至还有 L3；

但是这么多的告诉缓存，也带来了数据不一致的问题。那么久引入了内存模型 JMM。它是一种规范：规定了内存之间共享变量的读写问题。它只是一个理论，实际上解决这个缓存不一致问题的，有两种方案：

1：总线锁

在早期，人们解决这问题是在总线上加锁，保证同一时间只能由一个cpu 访问总线数据，为啥这样做呢，因为CPU个其他元件交互得通过总线。但是这样效率低啊，于是有了第二种方案。

1：总线锁导致总线风暴

锁总线的方案，是使用总线嗅探机制来检测是否有其他 CPU 修改了变量的值，如果有：就将缓存的值更新成新的值。

但是，多个线程频繁的读写操作，会造成大量的总线通信，引发总线风暴。降低了性能。

总线嗅探就是监视总线上的数据传输，以便于了解它传输的数据是否跟自己有关。

2：缓存一致性协议

最著名的是 Intel 的 MESI 协议。

它的核心思想是：当CPU在操作变量时候，发现它操作的是一个共享的变量，那么它会发出信号：其他CPU的缓存中失效，得从主内存取最新的值。

3：volatile 有序性

volatile 可以进制CPU指令重排序，它是通过内存屏障来实现。这保证了代码会严格按照顺序执行。

如：单例模式中的双重检查锁，就是用 volatile 防止指令重排序。

1：内存屏障

它是一种CPU 指令，是 JVM 在java 的class 文件的某些位置，插入这些指令，防止指令重排序。

具体是：

vloatile 写之前：加上 storestore 指令和 loadstore 指令；

volatile 写之后：加上 storeload 和 storestore

volatile 读之后：加上 loadload 和 loadstaore 。

各个操作系统上实现的方式还不一样。

4：JMM相关

1：有了 MESI 为啥还要 JMM

MESI 和 JMM 都是为了解决缓存一致性问题的，但是它们工作的范围和手段不一样。

MESI 是硬件层面的，JMM 是java 软件层面提出的一套规范，而且JMM规范的内容和解决的问题更广。

它们是不同维度的解决方案。

2：JMM

Java 内存模型。

它屏蔽了各种硬件和操作系统的访问差异，规范了工作内存和主内存交互的细节，也规范了如何解决内存有序性。

它不仅有一套规范，还提出了一些java 处理并发的关键字：sync 锁、volatile、fina、concurrent 包，供程序员使用。

现在说的 JMM, 一般都是 JDK 5 开始的新的 JMM。

他具体规范是：

​ 所有共享变量都存在主内存，各个线程有自己的工作内存，工作内存是主内存数据的一个副本，对共享变量只能在自己的工作内存操作，各个线程之间的工作内存独立且不能相互访问。

5：volatile 不能保证 i++ 的原子性操作

i ++ 操作分为三步骤：

  • CPU 拿到主内存的原始值到工作内存
  • 加操作
  • 把工作内存数据写回主内存

那就需要用到原子工具类、或者 sync 锁

6：为啥单例模式需要 volatile

1：new 一个对象的三步骤

  • 给对象分配内存空间
  • 对变量属性初始化
  • 将变量的引用地址写到内存

这就是为啥单例双重检查锁中，需要volatile 禁止指令重排序。

线程池是一种池化技术，所谓池化技术，就是提前保存大量的资源，以备不时之需；在机器资源有限的情况下，使用池化技术，有效提高资源利用率，提升性能；

线程池，就是提前创建好一批线程，保存在线程池中，当有任务需要执行时，就从线程池中选一个线程去执行，执行完毕再放回去，供其他任务使用；线程池的好处是它能控制并发，而且复用，减少创建和销毁的时间。

2：工作流程

2.1：也叫提交一个任务的流程

当提交一个任务时，先看看核心线程数有没有达到上限；

核心线程数没达到上限，就创建一个新核心的来执行任务；

如何核心满了，就把任务放到等待队列；

如果队列满了，就创建非核心线程从队列中取任务执行；

如果最大的线程数满了，新任务要触发拒绝策略；

当非核心线程数没有处理任务，超过它的空闲时间，就会被销毁。

注意：核心线程也是可以被销毁的。

前 5 个使用核心线程执行；

6-105 （100个）放入队列；

106-150（45个）非核心去执行；

1：要考虑几个维度

线程数的设定需要根据程序的需求和运行环境来设置，没有一个固定值；

首先看影响线程池因素有哪些：

  • CPU核心数

    • 多核处理器
    • 超线程技术（英特尔的HT技术），目前，很多CPU都采用超线程技术，也就是利用特殊的硬件指令。把两个逻辑内核模拟成两个物理芯片，让单个处理器都能使用线程级并行计算。所以我们经常看到 4核8线程CPU，就是物理内核4核，逻辑内核8核。可以为每个核心分配更多的线程，因为超线程可以提高CPU利用率。

  • 应用类型

    • CPU密集型

      主要是更多的计算，线程数的公式是：核心线程数 * 1 或者 1.5 ；因为这些主要是消耗 CPU，线程的上下文切换反而影响性能

    • IO密集型

      IO密集型主要涉及到很多的阻塞、等待（比如：数据库操作、文件操作、网络操作）；这样的看可以配置更多的线程，

      设置为 2 倍核心，因为线程需要更长的时间去处理更多的任务。

  • JVM 和内存资源

    • 内存大小
    • 操作系统的线程大小限制

  • 其他考虑

    • RT 要求（响应时间的要求）

      需要更多的线程

    • 任务类型的不同

      长时间的任务、短时间的任务、同步任务或者异步任务

  • 线程运行总时间

4：不建议直接套公式

我们有时候看到的 CPU 核心数并不一定是真实的；

很多时候，项目部署在云服务器上，有时候分配的机器显示的是8核，但是实际上使用的是虚拟机，并不是物理机；实际上并不能发挥出8核的作用；

可以在刚上线的时候，现根据公式设置一个值，然后再根据实际业务，以及不断的压测，再不断调整；最终达到一个合理的值。

然后在压测的时候，也要说清楚压测可接受的响应耗时是多少，大于这个阈值是错误的，错误率应该多少能接受。这样给出的线程池数量才是合理的。

6：如何直到线程池中所有线程已经执行完毕

不管是线程池内部还是外部，想要直到线程池中一个线程是否执行完毕，我们必须要获取线程执行结束后的状态，而线程本身是没有返回值的，所以只能通过阻塞、唤醒的方式来实现；

1：在内部咋知道

​ 线程池中的工作线程，是通过同步调用线程的 run 方法，等待 run 方法返回后，再去统计线程的完成数量；

2：在外部咋知道

​ 第一种：countDownLatch 计数器；

​ 第二种：线程池的 submit 方法会返回 future()，如果future 没有返回结果会一直阻塞，可以通过 future.get 判断线程是否执行完毕

7：如何让一个任务不进入阻塞队列，而是直接触发非核心线程去执行任务

​ 将线程池中的阻塞队列换成 SynchroQueue，这个队列啥也不存，没生产一个任务就会启动一个消费者去执行，否则就会阻塞生产者；

8：核心和非核心线程空闲时的回收

首先说线程池中核心线程和非核心线程都能被回收，但是一般不会回收核心线程，因为我们用线程池就是为了复用。

  • 核心线程在任务空闲的时候是阻塞的，不会占用CPU，调用的是 Queue的take方法；s

  • 非核心线程会被回收，因为调用的是 Queue的pull 方法；如果poll 方法返回null，代表空闲，非核心就要被回收

9：为啥核心线程数满了，不是直接调用非核心去执行任务，而是放入队列

这样做是为了避免核心线程执行的快，频繁的创建过多的非核心造成资源的浪费，而且还得回收，比较耗时；只有当队列满了，说明此时执行任务的压力比较大，才去使用临时工。

10：为啥不建议用内置线程池

因为线程池工具去创建，不能控制资源，没有指定队列长度、拒绝策略，如果请求过多会造成内存溢出。

11：线程池中的线程出现异常咋办

出现异常时，默认会把异常往外抛出；同时这个工作线程会销毁。

我们需要自己去设置应对异常的方式：

  • 对放到线程池中的任务要做异常捕获
  • 使用自定义线程工厂，设置异常处理机制

12：spring 线程池和 JDK 线程池的选择

  • jdk 线程池是 threadpoolExecutor
  • spring 线程池是 threadPoolTaskExecutor

spring 提供的线程池封装了 jdk 原生的线程池。

它提供了更多方便：

如控制生命周期、优雅关闭线程池。
